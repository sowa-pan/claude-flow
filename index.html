はじめに

本資料のゴール

Claudeを「チャットで質問するツール」から「制作フローの基盤」へ。AIに原稿を書かせることが目的ではなく、工程の下ごしらえと検品をAIに任せ、人間が編集判断に集中できる体制をつくる。

導入で変わること

工程 導入前 導入後
取材準備（リサーチ＋質問作成） 2〜3時間 20〜30分。生成＋人間が調整
文字起こし→内容整理 専用ツールで文字起こし 音源を入れるだけで自動処理
初稿 3〜6時間 2〜3時間。浮いた時間は編集へ
校正 通読を数回 観点別の自動チェック＋人間の最終判断
媒体ルールの継承 担当者の頭の中 レファレンスとして蓄積・共有

大前提

  • 書くのはAIでも、決めるのは人間
  • 何を残し、何を削り、どう並べるか。編集判断が品質の源泉であることは変わらない
  • AI導入は、編集判断に使える時間を増やすための投資

1. 3つのClaudeを使い分ける（Claude.ai / Cowork / Claude Code）

Claudeには3つの入口がある。中身は同じAIでも、入口ごとに使える道具と得意分野が変わる。一人の人物が場面ごとに違う顔を見せる、多重人格のようなイメージ。

1-1. 3製品の比較

Claude.ai Cowork Claude Code
形態 Web・スマホアプリ デスクトップアプリ ターミナル（コマンド画面）
主な機能 チャット、壁打ち、調べもの、プロジェクト機能 ファイルの読み書き、スキル実行、外部ツール連携、定型業務の自動化 Coworkと同等の機能＋開発向け機能。画面なしの自動実行も可
ファイル操作 添付したファイルのみ フォルダ単位で読み書き可能 同左。より広い範囲を扱える
向く業務 企画の壁打ち、リサーチ、単発の文章相談 原稿制作、文字起こし処理、書類の自動生成 大量ファイルの一括処理、定期バッチ、開発業務
対応環境 ブラウザがあれば可 Mac / Windows Mac / Windows

1-2. 使い分けの判断基準

  • 判断基準はファイルを扱うかどうか
    • ファイルの読み書きが発生する作業（原稿・文字起こし・画像・書類）→ Cowork
    • 頭の中の整理（企画・壁打ち・調査・相談）→ Claude.ai
  • 制作フローの自動化はCoworkで構築する
  • 計画はCowork、実行はClaude Codeという組み合わせも有効。スキルづくりや環境構築をCoworkで設計し、重い一括処理をClaude Codeで回す使い方
  • Claude CodeはCoworkの上級版という位置づけ。まずCoworkに慣れ、物足りなくなったら検討する

1-3. Claude Codeの使い所と使い方

エンジニア専用と思われがちだが、実体は「画面を持たないCowork」。チャット画面の代わりにターミナルで対話する。

使い方の基本。

  • ターミナル（Macはターミナル.app、WindowsはPowerShell等）でclaudeと入力して起動
  • あとはCoworkと同じく日本語で指示する。プログラミングの知識は必須ではない
  • CLAUDE.md（グローバル指示）とスキルは、CoworkとClaude Codeが共通の基盤（~/.claude）を土台にするため共用できる。ただし自動で完全同期されるわけではなく、特にスキルは保存場所をそろえる／同期する運用にしておくと確実（同期は週次のスケジュールタスクにもできる）

制作業務での使い所。

使い所 内容
大量ファイルの一括処理 数百ファイルのリネーム・形式変換・テキスト抽出など、画面表示が不要な力仕事
画面なしの自動実行 指示をコマンドとして渡し、結果だけ受け取る使い方。定期バッチに向く
バージョン管理との連携 Gitと組み合わせ、原稿の変更履歴を厳密に管理する運用
開発者との協業 制作チームのスキル・設定を開発者と共有し、社内ツール化する入口
  • トークン効率はCoworkより良い場合が多い。画面描画分のオーバーヘッドがないため
  • 導入順序の推奨：Claude.aiで慣れる → Coworkでフローを組む → 必要が出たらClaude Code

2. 自分と自社をAIに教える（基本設定・CLAUDE.md）

導入後、最初に行うべき作業。プロンプトの工夫より先に、自分が何者でどんなルールで仕事をしているかを設定として書く。一度書けば毎回のやりとりが短く正確になる。設定は「指示の貯金」。

2-1. 設定の3層構造

設定は効く範囲ごとに3つの層に分かれる。

レイヤー 場所 効く範囲
Instructions for Claude（旧カスタム指示） 設定画面（アカウント全体） Claude.aiのチャットとCoworkの両方。一度書けば両方に効く
グローバル指示（CLAUDE.md） 設定→Cowork→グローバル指示 Cowork・Claude Codeでのファイル作業すべて
プロジェクト指示 Claude.ai／Coworkのプロジェクト設定 そのプロジェクト内のみ
  • ポイントは、Instructions for Claudeがアカウント全体に効くこと。チャットとCoworkに同じ内容を二重に書く必要はない
  • CLAUDE.md（グローバル指示）は、Coworkでファイルを扱う作業に効く追加レイヤー。Claude Codeとも共通で使える
  • 優先順位は、固有で具体的な指示ほど優先される。プロジェクト指示（案件・媒体固有）＞全体に効く設定、の順で考える

2-2. 設定の構成例

実際に使っているCLAUDE.mdをそのまま例にする。媒体名や案件名は入れず、仕事の進め方と文体のルールを6項目で書いている。

# About Me
- 職業：ライター・編集者
- 使用ツール：Word、Excel、PowerPoint、Figma、VS Code

# Working Style
- 日本語で応答する（英語で聞かれた場合を除く）
- 情報が不足・曖昧な場合は推測せず確認する。事実と推測は明示的に区別する
- 複数の解釈ができる指示は、解釈を提示してから承認を待つ
- 選択肢を提示するときは AskUserQuestion ツールを使う
- 評価は良い点・悪い点を客観的に伝える
- 原稿作業の開始前に確認する：媒体名・文字数目安・トンマナ指定の有無

# Brand Voice
- 媒体指定がある場合はそちらのトンマナ優先
- ですます調では文末を敬体か体言止め中心にし、常体は少なめに抑える
- 同じ文末表現を連続させない。体言止め・倒置・言い換えでリズムをつける
- 見出し・本文に「——」を多用しない
- 根拠なく「興味深い」「重要な示唆」などの評価語を使わない
- 「結論から言うと」などの前置き宣言を入れない
- 締めの定型句（参考になれば幸いです、など）を入れない
- 「」（）を多用しない。補足は本文に組み込む
- 文字数目安が指定されていれば±10%以内に収める

# Design
- シンプル・ミニマル優先。装飾より余白と構造で読みやすさを出す
- 表は必要なときだけ使う
- Word文書の見出し階層はH1・H2中心

# Thread Management
- スレッドが重くなったり複数案件が混在したら「そろそろスレッドが重くなってきました。新しいスレッドに移りましょうか？」と声をかけ、引き継ぎメモを作成する

# Prompt Technique
- スキル修正・改善依頼は2段構成で対話する
  1. 問題点の指摘：現状の何が問題か・なぜ変えるかを1〜2文で確認する
  2. 改善案への問い返し：「では〜という方向でよいですか？」と方向性を提示し、承認を得てから実装する
- この2段構成で認識ずれを事前につぶし、往復回数を減らす

## fable モデル使用時のトークン節約ルール
- タスクをフェーズ分割してサブエージェントに委譲する
- fable：執筆・構成判断・設計／Sonnet 4.6：QC・Web検索・調査・構造化
- コンテキストは必要最小限に絞り、該当セクションだけ渡す
- 中間成果物はファイルに保存し、ファイル経由で次のサブエージェントへ引き継ぐ

2-3. 各項目の役割と効果

項目 書く内容 効果
About Me 職業、担当媒体、使用ツール 説明の前提・粒度が適切になる
Working Style 仕事の進め方、確認のルール 勝手な推測・先走りを防ぐ。事故防止に最も効く
Brand Voice 文体、表記、禁止表現 文章系アウトプットの手戻りが減る
Design 資料作成時の方針 過剰装飾を防ぐ
Thread Management 長くなったスレッドの扱い 精度劣化の前に引き継げる
Prompt Technique 修正依頼のやりとりの型 認識ずれを事前につぶし、往復回数を減らす

2-4. 書き方のコツ

  • 禁止形より行動形で書く。「曖昧なら推測するな」より「曖昧なら確認してから進める」
  • 理由つきのルールは守られやすい。「削除の前は影響を説明する。誤削除を防ぐため」
  • 1項目1行。長文の心構えより、短い具体的なルールの列挙
  • カテゴリで分ける（About Me・Working Style・Brand Voiceなど）。探しやすく、効かせたい場面も整理しやすい
  • 具体例を添える。ルールは行動形のまま、「文末にリズムをつける（〜です。〜です。の連続を避ける）」のように例を1つ付けると伝わりやすい
  • 長くしすぎない。CLAUDE.mdは長さに関係なく毎回全文が読み込まれるが、長い（目安200行超）ほど文脈を圧迫して遵守されにくくなる。そもそも強制ではなく参考として効くため、短く具体的に書くほど従ってもらいやすい
  • 完成を目指さない。AIの返答に違和感が出るたびに1行追加する運用で育てる

2-5. 自社版に置き換える

同じ枠組みで「自社を教える設定」をつくる。チームで共通化する項目の例。

  • 会社概要：事業内容、主要クライアント（ダミー名で可）、チーム構成
  • 品質基準：納品物のチェック観点、レビュー体制、責任の所在
  • 表記ルール：社内標準の表記統一ルール（記者ハンドブック準拠など）
  • 納品ルール：ファイル形式、命名、納品フォルダ、期日の数え方
  • NG事項：扱わない領域、クライアントごとの注意点

最小スタートは3項目。About Me、Working Style、Brand Voiceだけ書いて使い始め、運用しながら追記する。

3. 作業に合わせてAIを切り替える（モデルの使い分け）

3-1. 基本方針

Claudeには性能の異なる複数のモデルがあり、作業ごとに切り替えられる。

モデル帯 例 任せる仕事 理由
上位モデル Opus 4.8、Fable 5 執筆、構成判断、企画設計、プロット作成 判断の質が問われる。やり直しコストが高い
軽量モデル Sonnet 4.6 誤字脱字チェック、表記ゆれ確認、Web検索、調査、文字起こしの構造化 手数が多く正解が明確。速度と量で勝負

3-2. トークンの感覚

トークンはAIの稼働量の単位。プランごとに利用上限があり、使い方次第で消費速度が大きく変わる。

  • 消費が早いもの：長文の読み書き、画像の処理、PowerPoint・デザイン系の生成、長いスレッドの継続
  • 消費が遅いもの：短いテキストのやりとり、mdファイルの処理
  • 上位モデルほど消費が早い。全作業を上位モデルで行うと上限にすぐ達する
  • 体感として、上位モデル（Opus系）は検討が丁寧な分だけ時間がかかり、スレッドも早く埋まりやすい。コーディングの性能は高い一方、文章表現のうまさは必ずしもそれに比例しない。構成・判断は上位モデル、重い調査・整形は軽量モデル、文章の最終的な磨きは人間、と役割を分けると安定する

3-3. 自動振り分け（サブエージェント委譲）

  • モデルの使い分けは設定に書ける。「校正と調査は軽量モデルに任せる」と書けば、Claudeが作業に応じて自動で振り分ける
  • メインのAIが作業を切り出して別働隊のAIに渡す仕組みをサブエージェントと呼ぶ
  • 大きな作業をフェーズ分割し、フェーズごとに適したモデルへ委譲する設計が、品質とコストの両立につながる

4. AIが迷わない名前をつける（フォルダ・ファイル・カレンダーの命名）

AI活用の成否を分けるのは、プロンプトよりも命名ルール。原則は「AIが読んで把握できる名前をつける」。人間どうしで通じる省略語はAIには通じない。

4-1. 案件フォルダの標準構成

WORK/
└── 2026_月刊ダミーズ_株式会社サンプル取材/
    ├── 1企画書・資料・質問票/
    ├── 2取材音源/
    ├── 3文字起こし・構造化ノート/
    ├── 4原稿/
    ├── 5画像/
    └── 6納品/
  • フォルダ名：年＋媒体名＋案件内容
  • 工程順に並べた定型構成で、どの案件でも同じ場所に同じものがある状態を保つ
  • 先頭の番号は必須ではないが、つけておくと並び順が工程順に固定され、AIにもどのフォルダが何かが伝わりやすい
  • この状態ができていると「サンプル社の概要を読んでプロットを作成」という一言で、AIが迷わず該当ファイルに到達する

4-2. ファイル名のルール

形式：[日付8桁]_[媒体名]_[取材先]_[種別].md

例
OK 20260601_月刊ダミーズ_サンプル社_文字起こし.md
OK 20260610_月刊ダミーズ_サンプル社_初稿_v02.md
NG 最終.docx fix2.docx 本当の最終.docx
NG MD様_原稿.docx（社内略称はAIに通じない）
  • 日付はYYYYMMDDの8桁で統一。並べ替えと期間指定が機能する
  • バージョンは_v01 _v02の連番。「最終」「fix」は使わない
  • 略称・愛称を使う場合は、設定に対応表を書いておく（例：MD＝月刊ダミーズ）

4-3. カレンダーの登録ルール

形式：【種別】媒体名_相手先

  • 【取材】月刊ダミーズ_サンプル社
  • 【MTG】クライアントA_定例
  • 【〆切】月刊ダミーズ_初稿

種別の識別子があると、AIは次の処理を自動でできる。

  • 今日の予定から取材だけ抽出して準備メモを作成
  • 今週の〆切を一覧化してリマインド
  • 取材予定と案件フォルダのひもづけ（フォルダ名と媒体名が一致しているため）

タイトルが「サンプル社さん」だけでは、取材か会食か判別できず自動処理が機能しない。

4-4. 中間生成物はmdファイルで統一

文字起こし、構造化ノート、プロットなどの中間生成物はMarkdown（.md）形式で残す。

mdファイルは、見出しや箇条書きを「#」「-」などの記号で表す軽量なテキスト形式。Wordのような装飾情報を持たず、中身は素のテキストに近い。

作業中にmdを使うとフロー全体が速くなる理由は次の3つ。

メリット 内容
軽い 装飾情報を持たないぶんデータ量が小さく（同じ内容のWordの数十分の一）、AIの読み込み・書き出しが速い。トークン消費も小さい
差分が見やすい 素のテキストなのでバージョン間の比較が正確に効く（8-5の差分チェックの前提）
互換性が高い どのAIでもテキストエディタでもWordでも開ける

Wordは納品物など最終形だけに使う。初稿の段階もmdで回し、納品のタイミングでWord化するほうがフロー全体は速い。

5. Claudeを外部ツールとつなぐ（コネクタ・MCP・API）

5-1. コネクタの概念

コネクタは、Claudeが外部ツールを参照・操作するための接続口。一度接続すれば「カレンダーを見て」「Slackに送って」が言葉だけで通る。

コネクタ 制作業務での主な用途
Googleカレンダー 予定の読み取り。取材やMTG準備・リマインドの起点
Googleドライブ 資料の参照・共有。バックアップ・共有の置き場
Gmail 取材依頼・確認メールの下書き。受信メールの検索・要約、ラベルでの整理にも使える
Notion 企画管理・ナレッジの参照
Slack 自動処理の完了報告、チーム連絡、リマインドの送信
Figma / Canva ラフ・デザインデータの連携
  • 接続は各ツールのアカウント認証で行う。Claudeにパスワードを渡す方式ではない
  • 接続範囲（読み取りのみか、書き込みも許可するか）は認証画面で確認できる
  • 具体的な活用例は8章（制作フロー）と10章（日々の運用）で後述

5-2. MCPとは

MCP（Model Context Protocol）は、AIと外部ツールをつなぐための共通規格。コネクタの中身はこのMCPでできている。

  • たとえるなら、AIと外部ツールをつなぐUSB規格。規格に対応したツールなら同じ方法で接続できる
  • 標準で用意されたコネクタ以外にも、各社が公開するMCPサーバーを追加して接続先を増やせる
  • 使い所
    • 標準コネクタにないツールをつなぐ（タスク管理、CMS、データベースなど）
    • 社内システムとの連携。自社でMCPサーバーを用意すれば、社内DBや進行管理システムもClaudeから参照できる
  • 接続先が増えるほど入力できる情報の範囲も広がるため、接続するツールは業務上必要なものに絞る

5-3. APIとは

API（Application Programming Interface）は、チャット画面を介さずプログラムから直接Claudeを呼び出す窓口。

  • チャットやCoworkが「人がAIと対話する」入口だとすれば、APIは「システムがAIを呼ぶ」入口
  • 使い所
    • 自社サービス・社内システムへのAI機能の組み込み（例：CMSに校正ボタンを付ける）
    • 大量の一括処理（例：過去記事1000本の表記チェックを夜間に自動実行）
    • 決まりきった処理をUIなしで回すバッチ業務
  • 料金は処理量に応じた従量課金で、チャットのプランとは別契約
  • 制作チームが直接触るものではなく、開発会社に依頼してシステム化する段階での選択肢として知っておけばよい

5-4. 3つの関係の整理

仕組み 役割 誰が使うか
コネクタ ClaudeからGoogleやSlackなど外部ツールを操作する 利用者全員。設定画面から接続するだけ
MCP コネクタを支える共通規格。接続先を拡張できる 接続先を増やしたい管理者・開発者
API プログラムからClaude自体を呼び出す システム開発者

5-5. 他のAI（Gemini・ChatGPT）との連携

Claudeを軸にしつつ、特定のサブタスクだけ別のAIに任せる構成も組める。実際にこの制作フローでも、文字起こしは音声に強いGeminiに任せている（8-3）。

項目 内容
できること 役割分担で弱点を補う。長尺音声の文字起こしや超長文の処理はGemini、別観点の壁打ちや画像生成はChatGPT、というように工程単位で振り分ける。Claudeがメインで段取りし、サブ処理だけ外部AIに渡す
やり方 各AIのAPIキーを取得し、Coworkのスキルやスクリプトから呼び出す。MCPサーバー経由でつなぐ方法もある。普段は「この工程はGemini」と決めてスキルに組み込んでおくと、意識せず使える
リスク データが外部AIに渡る点に注意。守秘義務・個人情報・クライアント情報の扱いは事前に確認する。各社で学習やデータ保持のポリシーが異なり、APIキーの管理、出力品質のばらつき、誤情報の混入も避けられない
コスト 各AIのAPIは使った分だけの従量課金で、Claudeのプランとは別契約。重い工程だけに絞れば費用は小さく抑えられる
  • まずはClaude単体で回し、明確に弱い工程（長尺の文字起こしなど）だけ外部AIに任せるのが現実的

6. 案件・媒体ごとに指示と資料をまとめる（プロジェクト）

Claude.aiのプロジェクトは、案件や媒体ごとに指示と資料をまとめておく機能。

6-1. できること

指示欄に媒体のトンマナ・表記ルールを書いておくと、プロジェクト内のチャットは毎回その指示が効いた状態で始まる。企画書・過去記事・議事録・取材メモなどを登録しておけば、毎回のアップロードが不要になり、媒体らしさや過去の経緯をふまえた返答になる。メモリ機能でセッションをまたいでも過去の文脈を保持し、スレッドも媒体・案件ごとに整理されて、過去のやりとりを探しやすい。

6-2. ベストな活用例

媒体やクライアントごとに1プロジェクトをつくり、指示欄にトンマナ・表記・NG事項、知識として過去の好評記事2〜3本と編集部からの指摘メモを入れておく。連載・シリーズは回をまたぐ設定や決定事項をプロジェクトに残し、次回はそれを前提に相談する。やりとりが続くクライアント案件は、議事録・確定事項を入れておくと前回どこで合意したかをその場で確認できる。ただし入れすぎると焦点がぼやけるので、古い資料は入れ替え、最新で評価の高いものに絞る（9章の媒体レファレンスと同じ考え方）。

6-3. 使い分けの目安

プロジェクトと通常チャットは会話を整理する単位、Coworkはファイルを扱う製品で、そもそもレイヤーが違う。まず扱う作業で製品（Claude.ai／Cowork）を選び、そのなかで整理の単位（通常チャット／プロジェクト）を選ぶ。

レイヤー 選択肢 使いどころ
製品 Claude.ai 企画の壁打ち・調べもの・単発の相談。ファイルを扱わない用途
製品 Cowork 原稿・文字起こしなどファイル操作をともなう制作
整理の単位 通常チャット 単発の質問・雑談
整理の単位 プロジェクト 継続する案件・媒体の相談。指示や資料を常に効かせたいとき。Claude.ai・Coworkのどちらでも使える

6-4. レファレンス・スキルとの違い

プロジェクトに入れる資料と、媒体レファレンス（9章）・スキル（7章）は役割が違う。混同しやすいので整理する。

役割 プロジェクトに入れる？
スキル トリガー語で自動起動。設定で有効化すれば全チャットから使える 入れない。入れても起動せず、ただのテキストになる
媒体レファレンス 執筆・校正スキルが直接読む制作の正本。ローカルの_referencesが正本 原則入れない。Claude.aiのチャットで壁打ちするときだけ最小限に
プロジェクトのファイル その会話で毎回効かせたい指示・資料（トンマナ・企画書・過去記事） これがプロジェクト本来の用途

自動化の本筋（Cowork・Claude Codeのパイプライン）は、レファレンスとスキルで回す。プロジェクトは会話の前提を整える機能、と切り分けると迷わない。

7. よく行う手順を覚えさせて自動で呼ぶ（スキルとパイプライン）

制作フローの核心となる仕組み。

7-1. スキルとは

スキルは、よくやる仕事の手順書をファイル化してClaudeに渡しておく仕組み。実体はMarkdownで書かれた指示書で、次の要素で構成される。

# スキル名：質問票作成
## いつ使うか（トリガー）
「質問項目を作成」「取材の質問を考えて」と言われたとき
## 手順
1. 企画書と取材先情報を読み込む
2. 媒体の特性に合わせて質問を15問前後作成する
3. 大質問→深掘り質問の2階層で構成する
## アウトプット
Word文書。クライアント共有を想定した体裁にする
  • トリガーの言葉を含む依頼をすると、Claudeが該当スキルを自動で読み込み、手順どおりに実行する
  • チャットで毎回長い指示を打つ代わりに、手順書を一度書いて保存しておく発想
  • チャット履歴と違い、スレッドが変わっても消えない。品質が安定する

▼ トリガーでスキルが起動した画面（plot-creator／diff-learning）

7-2. プロンプトとスキルの違い

プロンプト スキル
入力 毎回打つ 一度書いて保存
品質 打ち方次第でブレる 手順が固定され安定する
共有 個人のノウハウに留まる ファイルとしてチームに配布できる
改善 言い直すだけで蓄積しない 修正点を書き戻して育てられる

7-3. 何をスキル化するか

判断基準は次の2つ。

  • 毎回ほぼ同じ指示を打っているか
  • アウトプットの品質基準が決まっているか

スキル化している作業の例（実運用ベース）。

分類 スキルの例
取材前 取材先リサーチ、質問票作成、想定回答の生成
取材後 文字起こしの整形、構造化ノート作成、プロット作成
執筆 媒体別の原稿執筆、見出し・リード文の生成
校正 誤字脱字チェック、ファクトチェック、最終チェック
事務 議事録作成、領収書整理、提案書・企画書の生成
環境管理 スキル台帳の更新、差分学習ログのスキャン

原稿まわりに限らず、事務作業も同じ仕組みでスキル化できる。

7-4. パイプライン＝スキルをつなぐ進行表

スキルが増えると、どれをどの順番で使うかが問題になる。スキルを正しい順番で呼び出す進行表を、それ自体ひとつのスキルとして持つ。これがパイプライン。

7-5. スキルのつくり方

ゼロからファイルを書く必要はない。次の3つをClaudeに渡せば、Claude自身がスキルの形（トリガー・手順・アウトプット）に組み立てる。

渡すもの 例
やりたいこと 取材の質問票づくりを自動化したい
守ってほしいルール・制約（上限） 質問は15問前後、大質問→深掘りの2階層、Word出力
参考資料 過去につくった質問票、媒体プロファイル

手順は3ステップ。

  • 【投げる】 やりたいこと・制約・参考資料を渡し、「これをスキルにして」と頼む
  • 【構築】 Claudeがトリガー・手順・アウトプットの形に整えてスキルを生成する
  • 【動作確認】 実際の依頼で一度走らせ、ずれた点を伝えて直す。ここから下の育て方につながる

つくるときの原則。

  • 1スキル1目的。何でもできるスキルは動きが不安定になる
  • 長くしすぎない。手順は10ステップ以内が目安
  • トリガーの言葉を明確にする。複数スキルでトリガーが重複すると誤作動の原因になる

7-6. スキルの育て方

  • スキルの出力に毎回同じ手直しをしているなら、その直し方をスキル本文にルールとして書き足す。次からはAIが最初からその形で出す
  • 目安は「同じ直しが2回出たら反映」。気づいたときだけでよく、これを続けるとスキルが自分の仕事のクセに馴染んでいく

8. 原稿を制作する（制作フローの全体像）

8-1. 全体フロー

工程 AIがやること 人間がやること
取材前 リサーチ、質問案、想定回答の生成 質問の取捨選択、取材の山場の設計
取材後 文字起こし、構造化ノート、仮プロット 内容の確認
原稿作成 プロット案の作成、初稿執筆 切り口の入力、プロット承認、修正指示

全自動にしない。要所に人間の判断を挟む設計が品質を担保する。

8-2. 取材前の準備

取材前の準備は、質問項目づくりと取材準備メモづくりの2本立て。後者はスケジュールタスク（10章）で半自動化している。

質問項目をつくる

AIに任せる場合は、質問作成スキルを使う。企画概要・取材概要・取材対象者の情報を渡すと、媒体の特性に合わせた質問案をWordで出力する。使うときのポイントは次のとおり。

  • 大質問→深掘り質問の2階層で組ませると、当日の話の広げ方がつくりやすい
  • 出てくる質問案は網羅的。取材の山場をどこに置くかは人間が決め、取捨選択と並べ替えで整える

取材・MTGの準備メモを自動でつくる

取材だけでなく、MTGや打ち合わせの事前準備にも同じ仕組みが使える。スケジュールタスクが毎日決まった時刻に動き、次の流れで準備メモを用意する。

  • 読み取り：複数のGoogleカレンダーを横断して2日後の予定を確認し、タイトルに「取材」「MTG」「打ち合わせ」「インタビュー」を含む予定を拾う（4章のカレンダー登録ルールが前提）
  • 参照：該当する案件フォルダの事前資料（企画概要・質問項目・取材対象者情報）を読み込む
  • 保存・通知：取材準備メモをWord文書として案件フォルダの事前資料の場所に保存し、Slackに「2日後に取材予定」と通知する。同名ファイルがあればスキップ

取材準備メモに入る内容は次のとおり。

  • 予定の概要（日時・形式・場所／接続情報）。対面ならアクセス・乗換案内・推奨出発時刻、オンラインなら会議URL
  • 取材先の多角的リサーチ。企業情報・人物プロフィール・直近1年のニュース・業界動向・競合・口コミ・媒体の既出記事との照合の7軸を、各軸に出典付きでまとめる（公開情報がない軸は「公開情報なし」と明記）
  • フォルダにある質問項目をベースにした想定回答例と深掘りポイント

8-3. 取材後の自動処理

取材後の自動処理がつくる成果物は次の4つ。

  • 文字起こし：音声を逐語で書き起こしたテキスト（音声から始めるパターン②でつくる。①は文字起こし済みのテキストが入り口）
  • 整形済みトランスクリプト：フィラー（えー・あのー）や相槌を除き、話者ラベルと段落を整えて読みやすくしたドキュメント
  • 構造化ノート（取材）または議事録（MTG）：トピック×話者で要点を蒸留したもの
  • プロット（取材のみ）：構造化ノートをもとにした記事の構成案

入り口の違いで、2つのフローを使い分けている。

パターン①：Coworkで完結（既存テキストを処理）

入り口は、すでに用意してある文字起こしテキスト。Coworkのスケジュールタスクが毎朝、指定のフォルダ内を巡回し、未処理の文字起こしテキストを見つけると、整形→構造化（取材）または議事録（MTG）→プロット（取材）まで自動で仕上げる。設定はCoworkの画面から言葉で頼むだけで、プログラミング不要。まず試すならこちら。

実例：自動処理スキルの中身

自動処理の中身は「オーケストレーター＋部品スキル」の組み合わせ。パターン①は torimaterial-post-processing、パターン②は audio-transcriber がオーケストレーターとして、ファイルごとに部品スキルを順に呼び出す。呼び出す4スキル（整形・構造化／議事録・プロット作成）は両パターン共通。

下の折りたたみが、その実際のスキルファイル（プロンプト本体）。自分の環境で自動化するには、部品となるスキルを登録し、それを束ねるスケジュールタスクを登録する。

登録手順

  1. スキルを登録：参照スキル①〜④のテキストをコピーし、Coworkの設定画面でスキルとして貼り付けて有効化する
  2. タスクを登録：先頭のオーケストレーターをスケジュールタスクに登録する。「毎朝、未処理の取材素材を処理して」のように言葉で頼めば設定できる（10章）
  3. 調整：媒体名や固有名詞を自分の案件に合わせて書き換える
  4. 動かす：以降は毎朝自動で整形→構造化→プロットまで仕上がる。単発なら冒頭のトリガー（「プロットを作って」など）で呼び出す

スケジュールタスク（オーケストレーター）｜取材後の自動処理（torimaterial-post-processing）

---
name: torimaterial-post-processing
description: WORK内の全フォルダをスキャンし、未処理の文字起こし生データを拾って整形・構造化・議事録・プロット作成するバックアップタスク。毎朝自動実行。「取材素材を後処理して」「文字起こしを処理して」などのトリガーでも手動実行可。

あなたは取材・MTG素材の一次処理タスクです。WORK内の未処理の文字起こしファイルを定期的に拾い上げ、プロジェクトコンテキストを自動収集してから整形・構造化・議事録・プロット作成を行います。カレンダーは参照しません。

## 実行方針

- **モデル**：Opus（メインスレッドはOpus前提）
- **subagent分離**：各ファイルの処理は `general-purpose` subagentに分離する（Agent toolで `model: opus` 指定）。メインスレッドは「ファイル一覧の管理＋完了レポート出力」だけを担い、コンテキスト圧迫を回避する
- **subagentに渡す必須情報**：
  - 参照すべきスキルファイルの絶対パス（最初にReadさせる）
  - プロジェクトコンテキスト（STEP 1.5で収集した話者名・製品名・音声誤認識候補）
  - 媒体判定ルール（フォルダ名→媒体プロファイル）
  - 出力ファイル名と保存先
  - エラー時のフォールバック方針（記録して次へ進む）

---

## 目的

`~/Desktop/WORK/` 以下の全フォルダをスキャンし、**未処理の文字起こし生データ**（.txt または .md）が残っていれば処理します。

## STEP 1: 未処理ファイルのスキャン（メインスレッドで実行）

`~/Desktop/WORK/` 直下の全サブフォルダを再帰的に走査します。
以下の条件でファイルを収集します：

**スキャン対象フォルダ（いずれかを満たせばOK）：**
- フォルダ名に「文字起こし」を含む（例：`4文字起こし・構造化ノート`）
- フォルダ名に「音源」を含む（例：`3取材音源`）
- ファイル名に「文字起こし」または「Transcription」を含む `.txt` / `.md` ファイルが置かれている任意のフォルダ

**未処理の定義：**
- `.txt` または `.md` ファイルが存在する
- かつ、対応する処理済みファイル（`_整形済み.docx`）が **同フォルダ内またはプロジェクトフォルダ（2階層上まで）の任意のサブフォルダ内** に存在しない
- かつ、以下の**日付フィルタ**を満たす：
  - まず**当日（今日の日付）に更新されたファイル**を探す
  - 当日ファイルが0件だった場合のみ、**過去3日以内**に範囲を拡大して再スキャンする
  - （当日中に手動実行した場合は今日のファイルが対象になり、翌朝のスケジュール実行では当日0件 → 前日以前のファイルにフォールバックする）
- ただし、ファイル名に `_整形済み` / `_構造化` / `_議事録` を含むファイルは対象外（出力済みファイル）

未処理ファイルが1件もなければ「処理対象なし。タスク終了。」と出力して終了します。

---

## STEP 1.5: プロジェクトコンテキストの収集（subagent起動前に実行）

文字起こしファイルが見つかったら、同じプロジェクトフォルダ内にある**企画書・取材質問・ヒアリングシート**を読み込み、整形スクリプト用のCONFIGを自動生成します。

### 1.5-A: コンテキストファイルの探索

プロジェクトフォルダ（文字起こしファイルから2階層上まで）を再帰的にスキャンし、以下に該当するファイルを収集します：

- ファイル名に「企画書」「企画」「ヒアリングシート」「取材質問」「質問リスト」「質問項目」「媒体説明」「依頼書」を含む `.docx` / `.txt` / `.md`
- ファイルサイズが上限500KB以内のもの（大きすぎるファイルはスキップ）

コンテキストファイルが1件も見つからない場合は、config.json を空の内容で作成して次のSTEPに進みます（処理自体は止めない）。

### 1.5-B: LLMによるコンテキスト抽出

収集したコンテキストファイルを読み込み、以下を推定します。**推測が難しい項目は空のままでOK**。確実に読み取れる情報だけをCONFIGに含めます。

**① 話者名マッピング（speaker_map）**
- ANKERブラケット形式（`speaker1: [HH:MM:SS - HH:MM:SS]`）のみ必要
- その他のフォーマット（Gemini・録音アプリ・手打ち形式）では空でよい
- 企画書や取材質問の冒頭から「取材対象者」「インタビュイー」の氏名を読み取る
- インタビュアー（ライター）は `"Q"` で統一。取材対象者は `"氏名様"` で設定
- 何人いるかわからない場合は speaker1〜speaker3 程度まで設定しておく

**② 音声誤認識修正リスト（replacements）**
- 企画書・取材質問から**固有名詞**（サービス名・製品名・社名・人名）を抽出する
- 抽出した固有名詞ごとに、音声認識で誤変換されやすいパターンを推定する
  - 例：「会計クラウドPro」→ 誤認識候補「会計クラウドプロ」「かいけいクラウド」「宵会計」
  - 例：「クラウド会計」→ 誤認識候補「フリー」「クラウド会計サービス」
- 確信度が低い場合は候補を入れなくてよい（誤修正より未修正の方がマシ）

**③ トピック見出し（topic_timestamps）**
- この段階では空 `{}` のままでよい
- タイムスタンプは文字起こしを読まないと確定できないため
- 完了レポートに「取材質問の大項目：[質問リストの大見出し一覧]」を記載し、手動追加を促す

**生成するconfig.json の形式：**

```json
{
  "speaker_map": {
    "speaker1": "Q",
    "speaker2": "田中様"
  },
  "topic_timestamps": {},
  "replacements": [
    ["会計クラウドプロ", "会計クラウドPro"],
    ["かいけいクラウド", "会計クラウド"],
    ["フリー(?=[をのがはとで、。　]|$)", "クラウド会計"]
  ]
}
```

### 1.5-C: config.json の保存

生成したconfig.jsonをプロジェクトフォルダ直下（または文字起こしファイルと同じフォルダ）に `_auto_config.json` として保存します。ファイルは次のSTEP 2のsubagentから参照します。

### 2-2.5: QCパス（自動修正）

`_整形済み.docx` を生成した直後に、以下の手順で自動QCを実施します。

**目的：毎回手動で指摘していた「話者の誤帰属」「固有名詞の誤認識」「名前の不整合」をここで自動修正し、ユーザーが修正指示を出すステップをゼロにする。**

#### A. 読み込み

生成した `_整形済み.docx` を `mcp__Word__By_Anthropic___get_document_text` または python-docx で読み込み、段落一覧（話者ラベル行＋本文）を取得する。

#### B. 固有名詞チェック

STEP 1.5 で読み込んだコンテキストファイル（企画書・取材質問）に登場する固有名詞（サービス名・社名・人名）と整形済みテキストを照合し、まだ修正されていない誤認識パターンを探す。

- 例：コンテキストに「会計クラウドPro」があるのに本文に「会計クラウドProネクスト」「会計ネクスト」が残っている
- 例：コンテキストに「サンプル会計事務所」があるのに「サンプル会計」「サンプル」になっている

検出した誤認識は python スクリプト（`--replacements` 追加）で一括修正する。件数が少なければ python-docx で直接書き換えてもよい。

#### C. 話者帰属チェック

**前提**：ANKER の話者分離精度は高くないため、元 .txt の段階で話者の誤帰属が発生している。format_transcript.py はその誤帰属をそのまま引き継ぐ。QCパスでは「整形後の .docx が間違っている」のではなく「ANKER の出力に含まれていたエラーをここで修正する」という位置づけで対処する。

具体的に以下のパターンを探す：

1. **話者ラベルが `speaker1` / `speaker2` のまま** → SPEAKER_MAP の設定漏れ。コンテキストから実名を補完して修正する
2. **内容の不一致（コンテキスト参照が必須）** → コンテキストファイルで確認できる「インタビュアーの役割」と「取材対象者の役割」を基準に判断する
   - 例：「Q」ラベルの話者が自社業務・製品・実績を主語として長く語っている → 取材対象者の発言が誤帰属されている
   - 例：取材対象者のラベルが付いた発言が「〜について教えていただけますか」などの質問形式になっている → Q の発言が誤帰属
3. **相槌・短い応答の誤帰属** → 「はい。」「そうですね。」「ありがとうございます。」など相槌がある話者に紛れ込んでいる。前後のブロックに吸収する

#### D. 修正の適用

- 修正量が少ない（3件以下）：python-docx で .docx を直接書き換え（bold段落のテキストを変更）
- 修正量が多い（4件以上）：追加の `--replacements` / `--speaker-map` を付与して `format_transcript.py` を再実行し、.docx を上書き
- 確信度が低い箇所は修正せず、QCサマリに「未修正（要確認）」として記録する

#### E. QCサマリ

STEP 3 の完了レポートに以下を追記する：

```
  → QC修正：{修正件数}件
    - 固有名詞：{件数}件（例：「会計クラウドProネクスト」→「会計クラウドPro」×3）
    - 話者修正：{件数}件（例：speaker2[00:08:14] → 山本様）
    - 要確認（未修正）：{件数}件 → {内容を列挙}
```

修正件数が0件の場合は「QC：変更なし」と記録。

# Audio Transcriber

取材音声を Gemini API で文字起こしし、プロジェクトフォルダに保存する。

## 使い方

1. Finderのサイドバーから **`audio-inbox`** を開く
2. 取材音声ファイル（`.ogg` / `.mp3` / `.mp4`）をドロップする
3. 自動的に文字起こしが始まる（macOS通知で開始・完了を通知）

手動で実行したい場合は「audio-inboxの音声を処理して」と話しかける。

## 前提条件

- `GEMINI_API_KEY` 環境変数が設定済みであること
- `google-genai` パッケージがインストール済み（`pip install google-genai`）
- 音声のファイル名またはサブフォルダ名に「媒体名_識別子」を含めること（投入元から案件フォルダを特定する）

## フォルダ構造

```
~/audio-inbox/                 ← 音声ファイルの投入口（Finderサイドバーに登録済み）

~/Desktop/WORK/
├── YYYYMM_媒体名_識別子/
│   ├── 1事前資料・企画概要・質問項目/
│   ├── 3取材音源/             ← 処理後に音声をリネームして移動
│   ├── 4文字起こし・構造化ノート/  ← 文字起こし出力先
│   └── ...
└── ...
```

---

## ワークフロー

### Step 1: audio-inbox をスキャン

```bash
ls -la ~/audio-inbox/
```

対象ファイル：`.ogg` / `.mp3` / `.mp4`。

- ファイルが0件 → 「audio-inbox に音声ファイルが見つかりません」と報告して終了
- 複数件 → 最初のファイルを対象として処理を続行する（確認不要）

### Step 2: 案件フォルダと種別を特定（ファイル名／サブフォルダから）

カレンダーには依存しない（無人実行ではカレンダーMCPが使えないため）。音声の「投入元の名前」から案件フォルダと種別を決める。投入元の名前は、サブフォルダに入っていれば**親フォルダ名**、`audio-inbox` 直下なら**ファイル名**を使う。

**特定ロジック（コードで実行すること）：**

```bash
AUDIO_BASENAME=$(basename "$AUDIO")
AUDIO_PARENT=$(basename "$(dirname "$AUDIO")")

# 投入元の名前：サブフォルダなら親フォルダ名、直下ならファイル名（拡張子・先頭日付を除く）
if [ "$AUDIO_PARENT" != "audio-inbox" ]; then
  SOURCE_NAME="$AUDIO_PARENT"
else
  SOURCE_NAME=$(echo "$AUDIO_BASENAME" | sed -E 's/\.[^.]+$//; s/^[0-9]{4}-?[0-9]{2}-?[0-9]{2}[_-]?//')
fi

# 種別：接頭辞で判定（MTG系→meeting、それ以外→interview）
if echo "$SOURCE_NAME" | grep -qiE '^(MTG|会議|打ち合わせ|打合せ|定例|社内|meeting)'; then
  MTG_TYPE="meeting"
else
  MTG_TYPE="interview"
fi

# 識別子：接頭辞（取材_/MTG_ 等）を除いた残り
PROJECT_ID=$(echo "$SOURCE_NAME" | sed -E 's/^(取材|MTG|会議|打ち合わせ|打合せ|定例|社内|meeting)[_-]?//')

# WORKフォルダ（YYYYMM_媒体名_識別子）を識別子で照合
PROJECT_DIR=$(ls -d ~/Desktop/WORK/[0-9][0-9][0-9][0-9][0-9][0-9]_* 2>/dev/null | grep -F "$PROJECT_ID" | head -1)

# 取材日：ファイル名にYYYYMMDDがあれば採用、なければ音声ファイルの更新日
INTERVIEW_DATE=$(echo "$AUDIO_BASENAME" | grep -oE '[0-9]{8}' | head -1)
[ -z "$INTERVIEW_DATE" ] && INTERVIEW_DATE=$(stat -f "%Sm" -t "%Y%m%d" "$AUDIO")
```

> 投入のしかた（どちらでもよい）。
> - ファイル名に入れる：`取材_媒体名_識別子.ogg` ／ `MTG_媒体名_識別子.mp3`（先頭に `YYYYMMDD_` を付けてもよい）
> - サブフォルダに入れる：`~/audio-inbox/取材_媒体名_識別子/録音.ogg`

### Step 3: 特定結果を確定する

`PROJECT_DIR` から最終的な識別子を取り出して保持する。

```bash
PROJECT_ID=$(basename "$PROJECT_DIR" | sed -E 's/^[0-9]{6}_//')
```

保持する変数。
- `PROJECT_DIR=~/Desktop/WORK/YYYYMM_媒体名_識別子`
- `PROJECT_ID=媒体名_識別子`
- `MTG_TYPE=interview|meeting`
- `INTERVIEW_DATE=YYYYMMDD`（Step 6・7・8・9のファイル名に使用する）

**特定できなかった場合**（`PROJECT_DIR` が空、またはフォルダが存在しない）：無人実行では誤った案件フォルダへ書き込むのを防ぐため、`案件フォルダを特定できませんでした（投入元: $SOURCE_NAME）。ファイル名またはサブフォルダ名に「媒体名_識別子」を含めてください。` とログに記録し、**そのファイルの処理を中止**する（音声は移動・リネームしない。他に未処理ファイルがあれば次へ進む）。

### Step 4: 事前資料を読む

```bash
ls "$PROJECT_DIR/1事前資料・企画概要・質問項目/"
```

フォルダ内のファイルを読む。読める形式（.txt / .md）を優先。.docx は必要に応じて読む。
PDF は PDF MCPを使用する。

資料から以下を抽出する：

| 項目 | 用途 |
|------|------|
| インタビュイーの氏名・肩書き | プロンプトの話者情報 |
| インタビュアー名（ライター名） | プロンプトの話者情報 |
| 専門用語・固有名詞・企業名 | プロンプトの表記統一リスト |

資料が存在しない / 読めない場合 → 話者名のみユーザーに確認して続行する。

### Step 5: Geminiプロンプトを構築

以下のテンプレートに資料から抽出した情報を埋め込み、`/tmp/gemini_prompt.txt` に保存する：

```
以下の取材音声を文字起こししてください。

【話者情報】
- インタビュイー：[氏名・肩書き]（例：田中氏・株式会社〇〇 代表取締役）
- インタビュアー：Q（複数いる場合はQ1、Q2と区別する）

【出力ルール】

■ 逐語記録
- 発言は一言一句そのまま書き起こす。話し言葉を書き言葉に直さない
- 「〜じゃないですか」「〜なんですよ」などの口語表現はそのまま残す
- 発言の省略・圧縮・まとめは禁止。どんなに長くても全発言を書き起こす

■ 話者
- 話者が変わるごとに話者名だけの行を置き、発言内容は次の行から書く
- 同じ話者が続く場合は話者名を繰り返さない
- インタビュアーは「Q」で統一する（複数人の場合はQ1・Q2）
- 話者の判定に迷ったら [話者不明] と記載する。推測で割り当てない

■ フィラー除去（音声を聞いて文脈で判断する）
- えー、あのー、えっと、うーん、なんか などは除去する
- 相槌・あいづちの「うん」「うんうん」「はい」は原則除去する
- 「まあ」「そうですね」は意味がある場合は残し、単なる間合いなら除去する
- フィラーを除去した結果、文として不自然にならないよう前後を整える（内容は変えない）

■ 終了条件（必読）
- 必ず音声ファイルの末尾まで書き起こすこと。途中で終了しない
- インタビューの締め（「ありがとうございました」「以上です」等）や、話者が「だいたいカバーできた」「大丈夫です」などと言った後も、音声が続く限り書き起こしを続けること
- 締めの後に追加の質疑・補足発言がある場合も、すべて省略せず記録すること
- 書き起こし完了後、最後の行に必ず以下を追記すること：
  [書き起こし完了]

■ トピック見出し
- 話題が大きく変わった箇所に ## 見出し を入れる（例：## 創業の経緯）
- 見出しは話題を端的に表す名詞句にする。要約・箇条書きは書かない
- 見出し直後に話者行を置く（見出しと話者行の間に空行不要）

■ 聞き取り・固有名詞
- 聞き取れない箇所は [聞き取り不明] と記載する。推測で補わない
- 下記の固有名詞リストに近い音が聞こえた場合、文脈を判断して正式表記に置き換える
- 置き換えた箇所は変更しない（注記不要）

■ 要確認マーク
以下のいずれかに該当する箇所には、発言の直後に　〔要確認 HH:MM:SS〕　を挿入する。
HH:MM:SS は該当箇所の音声タイムスタンプ（例：〔要確認 00:12:34〕）。
－ 音声が不明瞭・音割れ・ノイズで聞き取りに自信がない
－ 話者の判定が不確かで、割り当てを間違えている可能性がある
－ 複数人が同時に話していて発言の区切りが不明確
－ 固有名詞・専門用語の音が曖昧で正確な表記に確信が持てない

【固有名詞リスト】
[資料から抽出した用語・固有名詞・企業名・サービス名を列挙]

【出力イメージ】
## 創業の経緯

氏名
発言内容がここに入る。話し言葉のまま、省略せず書き起こす。

Q
質問内容がここに入る。
```

# Transcript Formatter

文字起こしファイルをスクリプトで処理し、整形済みdocxを出力する。

## 対応フォーマット（自動判定）

| フォーマット | 拡張子 | 判定条件 |
|------------|--------|---------|
| ANKERブラケット形式 | .txt | `speaker1: [00:01:05 - 00:01:10]` 形式（コロン＋角括弧）。ファイル名に「Transcription」を含む場合（例：「会議名-Transcription.txt」）も本形式。冒頭のタイトル・日付行は自動スキップ |
| ANKER形式 | .txt | `speaker1 00:01:05` で始まるブロック（スペース区切り） |
| Gemini形式 | .txt | `[cite_start]` または `[cite: N]` マーカーを含む（Shift-JIS/UTF-8両対応） |
| 録音アプリ形式 | .txt | `話者名 01:05` で始まるブロック（MM:SS） |
| 録音アプリ形式 | .docx | 段落ごとに `話者名（改行）テキスト` |
| 手打ち形式 | .docx | `○話者名　テキスト`（全角スペース区切り） |
| Markdown見出し形式 | .md | `### 話者名 HH:MM:SS` で始まるブロック |

**判定順序：** Gemini → ANKERブラケット → ANKER → その他の順で判定する。Geminiは `[cite_start]` マーカーの有無で判別。ANKERブラケットはANKERより先に判定すること（どちらも `speaker` で始まるため）。

## 出力書式（固定ルール）

スクリプトが生成するdocxの書式は以下に固定されている。変更しないこと。

docx-formatスキルおよびdocxスキルに準拠。

### ページ設定

| 項目 | 設定値 |
|------|--------|
| 用紙 | A4 |
| 余白 | 狭い（12.7mm = 1.27cm）全辺 |
| フォント | Yu Gothic（10.5pt） |

### 段落書式

| 要素 | スタイル | 詳細 |
|------|---------|------|
| ドキュメントタイトル | **Normal**（Heading禁止） | bold 14pt、space_before=0、space_after=12pt。入力ファイル名から自動生成 |
| トピック見出し | **Normal**（Heading禁止） | bold 11pt、space_before=16pt、space_after=6pt |
| 話者名 | Normal | **独立段落**。bold 10.5pt。space_before=12pt、space_after=2pt |
| 本文段落 | Normal | 10.5pt。space_before=0、space_after=4pt |
| 同一話者内の1行アキ | Normal | **空行段落を挿入**（space_before=0、space_after=0）。space_afterで行間調整する方法は使用禁止 |

### フォント設定の注意

`font.name = "Yu Gothic"` はASCII（英数字）にしか効かない。日本語文字（East Asian）には
XMLレベルで `w:rFonts` の `w:eastAsia` 属性を直接設定する必要がある。
`set_run_font()` がこれを担当する。Normalスタイルにも同様の設定を行う。

### Headingスタイル禁止

`doc.add_heading()` および `style="Heading N"` は**使用禁止**。
Wordの環境によって「■」が自動付与されるため。
見出しはすべてNormalスタイル＋フォント装飾で再現する。

## 手順

### Step 1: スクリプトをコピー

```bash
cp <このSKILL.mdのディレクトリ>/scripts/format_transcript.py /tmp/format_transcript.py
chmod u+w /tmp/format_transcript.py
```

`/tmp/` はCowork・Claude Code両環境で使える共通パス。

### Step 2: CONFIG を編集

`/tmp/format_transcript.py` の上部にある CONFIG セクションを書き換える。

**SPEAKER_MAP（話者名マッピング）**

- ANKERフォーマットのみ必須（`speaker1` → 実名）
- Markdown形式で `speaker2` などの未識別話者がある場合も設定する
- Gemini形式は `**山本：**` などの話者名がそのまま使われる。マッピングが必要な場合のみ設定する
- それ以外のフォーマットは話者名がそのまま使われるので空でOK

```python
SPEAKER_MAP = {
    "speaker1": "担当者（サンプル社）",
    "speaker2": "Q",
}
```

**TOPIC_TIMESTAMPS（トピック見出し）**

- 基本的に設定する。タイムスタンプと見出し名をユーザーに確認して埋める
- **Gemini形式は不要：** `###` セクション見出しがそのまま反映される
- タイムスタンプがないフォーマット（手打ちdocx等）や、見出しが不要と明示された場合のみ空 `{}` のままにする
- ANKER形式・Markdown形式は `HH:MM:SS`、録音アプリ形式は `MM:SS` で記述

```python
TOPIC_TIMESTAMPS = {
    "00:00:00": "取材開始・進行確認",
    "00:07:10": "テーマ名",
}
```

### Step 3: スクリプトを実行

```bash
python3 /tmp/format_transcript.py INPUT_FILE OUTPUT.docx
```

入力ファイルは `.txt` / `.docx` / `.md` に対応。フォーマットは自動判定する。

### Step 3.5: 接続詞整流スクリプトを実行

スクリプト出力は話し言葉の接続詞連鎖（「で、ただ、でも、」等）が残る。
`step35_auto.py` で自動除去する。

```bash
cp <このSKILL.mdのディレクトリ>/scripts/step35_auto.py /tmp/step35_auto.py
python3 /tmp/step35_auto.py INPUT.docx OUTPUT_v2.docx
```

`step35_auto.py` はコマンドライン引数でINPUT/OUTPUTを受け取る。引数2つ指定が基本。引数1つの場合は自動的に `_v2` サフィックスで出力する。スクリプト内のINPUT/OUTPUT変数を直接書き換える必要はない。

スクリプトが自動処理する内容：

- 接続詞連鎖の解消（「で、ただ、」「で、でも、」「て、で、」等）
- 文中・文頭の余分な「で、」除去
- フィラー残り追加除去（えーと、えっと、あのー等）
- 読点の重複・句点前の読点除去

**処理対象：** 本文段落（非bold）のみ。話者名・見出しはスキップ。
**処理しない：** 語彙・表現の書き換え。ルールに該当しない箇所は原文保持。

後処理済みファイルはスクリプト出力のファイル名に `_v2` を付けて保存することを推奨。

### Step 4: 出力ファイルをユーザーに渡す

`computer://` リンクで `mnt/outputs/` 内の後処理済みファイル（`_v2`）を提示する。
スクリプト出力（元ファイル）も合わせて提示してよい。

## 処理内容

### format_transcript.py が行うこと

- フォーマット自動判定（5種類）
- フィラー除去（えー・あのー・えっと・まあ・なんか等）
- 短断片マージ（タイムスタンプがあるフォーマットで有効）
- 同一話者の連続ブロック結合
- 句点基準の段落分割（目標280字・上限450字）
- 話者名（独立段落・bold 10.5pt）＋本文段落（10.5pt）を別段落で出力
- 同一話者の発言が複数段落に分割された場合、段落間に空行段落を挿入（1行アキ）
- トピック見出しをNormalスタイル＋装飾で挿入（TOPIC_TIMESTAMPSが空なら非挿入）
- 入力ファイル名からドキュメントタイトルを自動生成（冒頭に配置）

### step35_auto.py が行うこと

- 接続詞連鎖の解消（ルールベース、約20パターン）
- フィラー残り追加除去
- 読点の重複・不正除去
- 語彙・表現は変更しない

# MTG Minutes

MTGの記録（文字起こし・メモ・Slack等）から、すぐに共有・保管できる議事録Word文書（.docx）を作成する。

## 目的

「誰が・何を・どう決めたか」を関係者全員が即座に把握できる議事録を作ること。

発言の再現ではなく**情報の蒸留**が目的。論点・決定事項・アクションアイテムが一目でわかる構成にする。

## 処理手順

### 1. インプット確認

まず以下を確認する：
- 参加者が特定できるか（不明な場合はユーザーに確認）
- MTGの目的・テーマ（わからなければ記録から推定し、ユーザーに提示して確認）
- 日時・場所（記録になければ「不明」と記載）

### 2. 内容の読み込みとトピック抽出

インプット全体を通読し、議題・話題の流れを把握する。

**トピック分割の基準：**
- 話題の転換点で区切る
- 1トピック = 1つの議題・テーマ
- トピック名は内容を端的に表す具体的なタイトルにする（「〇〇の進捗確認」「〇〇の方針決定」など）

### 3. 各セクションの抽出

以下の4セクションを抽出・整理する。

#### セクション①：アジェンダ・参加者

- 日時・場所
- 参加者（所属・役職があれば併記）
- アジェンダ（議題の箇条書き）

#### セクション②：議事要旨

トピックごとに、発言をラベル付きインデント形式で整理する（transcript-structurizerと同形式）。

**整理ルール：**
- 話者名 → ラベル付き発言要約の構造
- ラベルは2〜6字の名詞句（「現状報告」「課題提起」「提案内容」「懸念点」など）
- 発言をそのまま転記せず、要点を圧縮して表現する
- 印象的な言葉・キーワードは「」で原文引用
- 脱線・雑談はカット（カットした場合は注記を残す）
- 事実・数字・固有名詞は省略しない

#### セクション③：決定事項・アクションアイテム

**決定事項**（MTGで合意・確定した内容）：
- 箇条書きで列挙
- 「何が決まったか」を1行で端的に

**ToDo・対応事項**（今後やること）：
- 担当者・タスク・期限を箇条書きで整理（表は使わない）
- 形式：`・担当者　タスク内容　→ 期限`
- 期限が明言されていない場合は「要確認」と記載
- 担当者が不明な場合も「要確認」と記載

#### セクション④：次回MTG

- 日時（決まっていれば）
- 場所/形式
- 次回アジェンダ（言及があれば）

### 4. 出力

出力は**Word文書（.docx）**として作成する。**python-docxを使ったPythonスクリプトを生成・実行**すること（npm/docx-jsは使わない）。

#### 出力フォーマット（構造イメージ）

〔日時・場所・参加者などのメタ情報（グレー）〕

  ・〔議題1〕
  ・〔議題2〕

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

      ラベル1: 発言内容の要約
      ラベル2: 発言内容の要約「印象的な表現は原文引用」

#### Pythonスクリプトのテンプレート

以下の関数構成でスクリプトを生成・実行する：

```python
#!/usr/bin/env python3
# -*- coding: utf-8 -*-
from docx import Document
from docx.shared import Pt, Mm, RGBColor
from docx.oxml.ns import qn
from docx.oxml import OxmlElement
import os

FONT_NAME = "Yu Gothic"
OUTPUT_PATH = "/mnt/user-data/outputs/[ファイル名]_議事録.docx"

def set_rfonts(run):
    rPr = run._r.get_or_add_rPr()
    rFonts = OxmlElement('w:rFonts')
    rFonts.set(qn('w:eastAsia'), FONT_NAME)
    rPr.insert(0, rFonts)

def set_font(run, size_pt=10, bold=False, color=None):
    run.font.name = FONT_NAME
    run.font.size = Pt(size_pt)
    run.font.bold = bold
    if color:
        run.font.color.rgb = RGBColor(*color)
    set_rfonts(run)

def set_narrow_margins(doc):
    for section in doc.sections:
        section.page_width = Mm(210)
        section.page_height = Mm(297)
        section.top_margin = Mm(12.7)
        section.bottom_margin = Mm(12.7)
        section.left_margin = Mm(12.7)
        section.right_margin = Mm(12.7)

def set_default_font(doc):
    style = doc.styles['Normal']
    style.font.name = FONT_NAME
    style.font.size = Pt(10)
    rPr = style.element.get_or_add_rPr()
    rFonts = OxmlElement('w:rFonts')
    rFonts.set(qn('w:eastAsia'), FONT_NAME)
    rPr.insert(0, rFonts)

def add_title(doc, text):
    p = doc.add_paragraph()
    p.paragraph_format.space_before = Pt(0)
    p.paragraph_format.space_after = Pt(4)
    r = p.add_run(text)
    set_font(r, size_pt=14, bold=True)

def add_meta(doc, text):
    p = doc.add_paragraph()
    p.paragraph_format.space_before = Pt(0)
    p.paragraph_format.space_after = Pt(6)
    r = p.add_run(text)
    set_font(r, size_pt=9, color=(100, 100, 100))

def add_section_heading(doc, text):
    """■ セクション見出し"""
    p = doc.add_paragraph()
    p.paragraph_format.space_before = Pt(10)
    p.paragraph_format.space_after = Pt(4)
    r = p.add_run(text)
    set_font(r, size_pt=12, bold=True)

def add_divider(doc):
    """区切り線"""
    p = doc.add_paragraph()
    p.paragraph_format.space_before = Pt(6)
    p.paragraph_format.space_after = Pt(6)
    r = p.add_run("─" * 40)
    set_font(r, size_pt=9, color=(180, 180, 180))

def add_h2(doc, text):
    """トピック見出し（H2相当）"""
    p = doc.add_paragraph()
    p.paragraph_format.space_before = Pt(10)
    p.paragraph_format.space_after = Pt(3)
    r = p.add_run(text)
    set_font(r, size_pt=11, bold=True)

def add_bullet(doc, text, indent_level=1):
    """箇条書き（セクション内）"""
    p = doc.add_paragraph(style='Normal')
    p.paragraph_format.space_before = Pt(1)
    p.paragraph_format.space_after = Pt(1)
    p.paragraph_format.left_indent = Pt(14 * indent_level)
    r = p.add_run(f"・{text}")
    set_font(r, size_pt=10)

def add_speaker(doc, speaker):
    """話者名（第1レベル）"""
    p = doc.add_paragraph(style='Normal')
    p.paragraph_format.space_before = Pt(4)
    p.paragraph_format.space_after = Pt(1)
    p.paragraph_format.left_indent = Pt(14)
    r = p.add_run(speaker)
    set_font(r, size_pt=10, bold=True)

def add_item(doc, label, content):
    """ラベル: 内容（第2レベル）"""
    p = doc.add_paragraph(style='Normal')
    p.paragraph_format.space_before = Pt(1)
    p.paragraph_format.space_after = Pt(1)
    p.paragraph_format.left_indent = Pt(28)
    if label:
        r_label = p.add_run(f"{label}: ")
        set_font(r_label, size_pt=10, bold=True)
    r_content = p.add_run(content)
    set_font(r_content, size_pt=10)

def add_action_table(doc, actions):
    """アクションアイテム表。actionsは[(担当者, タスク, 期限), ...]のリスト"""
    table = doc.add_table(rows=1, cols=3)
    table.style = 'Table Grid'
    hdr = table.rows[0].cells
    for i, h in enumerate(["担当者", "タスク", "期限"]):
        hdr[i].text = h
        for run in hdr[i].paragraphs[0].runs:
            set_font(run, size_pt=10, bold=True)
    for person, task, deadline in actions:
        row = table.add_row().cells
        row[0].text = person
        row[1].text = task
        row[2].text = deadline
        for cell in row:
            for run in cell.paragraphs[0].runs:
                set_font(run, size_pt=10)

# ドキュメント生成
os.makedirs(os.path.dirname(OUTPUT_PATH), exist_ok=True)
doc = Document()
set_narrow_margins(doc)
set_default_font(doc)

# タイトル・メタ
add_title(doc, "〔MTGタイトル〕議事録")
add_meta(doc, "日時：〔日時〕　場所/形式：〔場所・オンライン〕\n参加者：〔参加者〕")

# アジェンダ
add_section_heading(doc, "■ アジェンダ")
add_bullet(doc, "〔議題1〕")
add_bullet(doc, "〔議題2〕")

add_divider(doc)

# 議事要旨
add_section_heading(doc, "■ 議事要旨")
add_h2(doc, "1. 〔トピック名〕")
add_speaker(doc, "〔話者名〕")
add_item(doc, "〔ラベル〕", "〔発言内容の要約〕")

add_divider(doc)

# 決定事項
add_section_heading(doc, "■ 決定事項")
add_bullet(doc, "〔決定内容1〕")

# ToDo・対応事項
add_section_heading(doc, "■ ToDo・対応事項")
add_action_item(doc, "〔担当者〕", "〔タスク〕", "〔期限〕")

add_divider(doc)

# 次回MTG
add_section_heading(doc, "■ 次回MTG")
add_bullet(doc, "日時：〔日時〕")
add_bullet(doc, "場所/形式：〔場所・オンライン〕")
add_bullet(doc, "次回アジェンダ：〔言及があれば〕")

doc.save(OUTPUT_PATH)
print(f"保存完了: {OUTPUT_PATH}")
```

**スクリプト生成時の注意：**
- 実行前に `pip install python-docx --break-system-packages` でライブラリを確認する
- 出力先パスはユーザーに確認するか、デフォルトで `/mnt/user-data/outputs/` に保存する
- ToDo・対応事項が0件の場合は「なし」と記載する
- 次回MTGが未定の場合は「未定」と記載する
- メタ情報はインプットから読み取れる範囲で記載し、不明な項目は「不明」とする

#### docxスタイル仕様

# Transcript Structurizer

インタビューや会議の文字起こしを**トピック×話者のインデント形式**で整理し、文字起こしを読まなくても内容を完全に把握できる構造化データ（.docx）を作成する。

## 目的

**文字起こしを読まなくても、すべての情報を把握できる構造化データを作ること。**

会話をそのまま再現するのではなく、**情報を蒸留する**ことが目的。各話者の発言を「何について、何を言ったか」というラベル付きインデント形式に変換し、内容を圧縮・整理する。

網羅性は保つ。重要でないと判断してカットしない（脱線・雑談は除く）。情報の重要度判断は後工程（plot-creator等）の仕事であり、本スキルでは行わない。

## transcript-formatterとの違い

| | transcript-formatter | transcript-structurizer（本スキル） |
|---|---|---|
| 目的 | 原文を読みやすく整形 | 内容を要約・蒸留して構造化 |
| 出力 | 整形済み会話テキスト | トピック×話者のインデント形式要約（.docx） |
| 原文との関係 | 一言一句保持 | 情報を圧縮・ラベル化して再構成（ただし網羅的に） |

ユーザーが「整形」「読みやすく」と言った場合 → transcript-formatter
ユーザーが「整理」「まとめて」「構造化」「要約して」「構造化データ」と言った場合 → 本スキル

## 処理手順

### 1. 事前確認——インタビューかミーティングかを判定する

文字起こしを受け取ったら、まず種別を確認する。

#### カレンダーで判定（推奨）

文字起こしファイルの日付（ファイル名の YYYYMMDD、または更新日時）を元に、Googleカレンダー MCP (`list_events`) でその前後3日間を検索する。

| イベントタイトルのパターン | 種別 |
|---|---|
| `【取材` を含む（`【取材online】`・`【取材対面】` 等） | **interview** → このスキルで続行 |
| 上記以外（`【MTG】`・`【定例】`・`【打ち合わせ】` 等） | **meeting** → `mtg-minutes` スキルに委譲して終了 |

#### カレンダーで判定できない場合（無人実行を含む）

カレンダーMCPが使えないときは、ファイル名・親フォルダ名のキーワードで判定する。

- 「MTG」「mtg」「打ち合わせ」「定例」「社内」「meeting」「ミーティング」を含む → **meeting**（`mtg-minutes` に委譲して終了）
- 含まなければ **interview**（既定。このスキルで続行）

対話実行で曖昧なときのみ、ユーザーに「インタビュー・対談ですか、ミーティングですか？」と確認してよい。無人実行では確認せず上記の既定で進める。

判定後、以下も確認する：
- 話者が特定できるか（話者名が不明な場合はユーザーに確認）

### 2. 通読・トピック抽出

文字起こし全体を通読し、主要トピックを洗い出す。

**トピック分割の基準：**
- 話題の転換点で区切る
- 1トピック = 1つの主題（細かすぎず、大きすぎず）
- 時系列で自然に並ぶ場合は時系列順、テーマが明確な場合はテーマ別にグルーピング
- トピック名は内容を端的に表す具体的なタイトルにする（「〇〇について」「〇〇の経緯」など）

### 3. 話者ごとの発言を要約・インデント形式に変換

各トピック内で、話者ごとに発言内容を**ラベル付きインデント形式**に変換する。

**整理のルール：**

- **ラベル付きインデント**：各発言を「**ラベル**: 内容の要約」という形式に変換する。ラベルは2〜6字程度の名詞句が理想（例：「採用方針」「転機の理由」「バイアスへの対処」「心理的安全性」）
- **情報の圧縮**：発言をそのまま転記するのではなく、要点を抽出して簡潔に表現する。1つの発言ラベルで1つの情報単位（主張・事実・エピソード）を表す
- **長い発言は分解**：複数の情報が含まれる発言は、複数の発言ラベルに分解する
- **印象的な表現は原文引用**：比喩・キーワード・感情が込められた言葉は「」で原文のまま残す（例：「父の言葉: 「3年は続けろ」が転機に」）
- **質問（インタビュアー）の扱い**：インタビュアーの質問は、回答者のブロックに含めず、**`Q：〔質問の要点〕` の1行形式**で記録する。「Q：」部分のみbold、質問内容は通常ウェイト。話者ブロック＋ラベル行の2行構成にはしない。長い質問も要点のみに圧縮してよい。質問がない箇所は省略する
- **具体例・エピソードは保持**：数字・固有名詞・エピソードは省略しない。ただし要点に絞って記述する
- **捏造禁止**：文字起こしにない情報を追加しない。推測や解釈を加えない
- **脱線・雑談のカット**：本筋から逸れる会話や明らかに関係のない雑談はカットしてよい。カットした場合は「※このあたりで〇〇に関する雑談あり（省略）」と注記を残す
- **網羅性**：上記の範囲内でカットせず、文字起こしの全トピック・全発言ラベルを反映する

**発言の粒度：**
- 1つの発言ラベル = 1つの情報単位（主張・事実・エピソード）
- ラベルは短く、内容は1〜3文程度に圧縮する

### 4. 出力

出力は **Word文書（.docx）** として作成する。**python-docxを使ったPythonスクリプトを生成・実行**して生成すること（npm/docx-jsは使わない）。

#### 出力フォーマット（テキスト構造イメージ）

```
〔文書タイトル〕
〔取材日・媒体・対象者・聞き手などのメタ情報（グレー）〕

1. 〔トピック名〕

  Q：〔質問の要点〕（「Q：」のみbold、インデント第1レベル）

  話者A（bold、インデント第1レベル）
      ラベル1: 発言内容の要約。具体的な数字・固有名詞を含む
      ラベル2: 発言内容の要約。「印象的なフレーズは原文のまま引用する」

  話者B（bold、インデント第1レベル）
      ラベル1: 発言内容の要約

2. 〔トピック名〕

  Q：〔質問の要点〕（「Q：」のみbold、インデント第1レベル）

  話者A（bold、インデント第1レベル）
      ラベル1: 発言内容の要約

（以下、トピックの数だけ繰り返す）
```

**フォーマット補足：**
- トピック名は番号付き見出し（H2レベル、12pt bold）にする
- 話者名は第1レベルのインデント（10pt bold）
- 発言ラベルは第2レベルのインデント（ラベル部分のみbold、コロン以降は通常）
- インタビュアーの質問は `Q：〔質問の要点〕` の1行形式で、回答者ブロックの前に配置する
- 質問がない箇所はインタビュアーのブロックを省略する
- 1トピック内で質問→回答が複数ある場合は、その都度 `Q：〔質問〕` → 回答者ブロックを繰り返す
- 回答者の話者名は文字起こし内の表記に合わせる

#### Pythonスクリプトのテンプレート

以下の関数構成でスクリプトを生成・実行する：

```python
#!/usr/bin/env python3
# -*- coding: utf-8 -*-
from docx import Document
from docx.shared import Pt, Mm, RGBColor
from docx.oxml.ns import qn
from docx.oxml import OxmlElement
import os

FONT_NAME = "Yu Gothic"
OUTPUT_PATH = "/mnt/user-data/outputs/[ファイル名]_構造化データ.docx"

def set_rfonts(run):
    rPr = run._r.get_or_add_rPr()
    rFonts = OxmlElement('w:rFonts')
    rFonts.set(qn('w:eastAsia'), FONT_NAME)
    rPr.insert(0, rFonts)

def set_font(run, size_pt=10, bold=False, color=None):
    run.font.name = FONT_NAME
    run.font.size = Pt(size_pt)
    run.font.bold = bold
    if color:
        run.font.color.rgb = RGBColor(*color)
    set_rfonts(run)

def set_narrow_margins(doc):
    for section in doc.sections:
        section.page_width = Mm(210)
        section.page_height = Mm(297)
        section.top_margin = Mm(12.7)
        section.bottom_margin = Mm(12.7)
        section.left_margin = Mm(12.7)
        section.right_margin = Mm(12.7)

def set_default_font(doc):
    style = doc.styles['Normal']
    style.font.name = FONT_NAME
    style.font.size = Pt(10)
    rPr = style.element.get_or_add_rPr()
    rFonts = OxmlElement('w:rFonts')
    rFonts.set(qn('w:eastAsia'), FONT_NAME)
    rPr.insert(0, rFonts)

def add_title(doc, text):
    p = doc.add_paragraph()
    p.paragraph_format.space_before = Pt(0)
    p.paragraph_format.space_after = Pt(4)
    r = p.add_run(text)
    set_font(r, size_pt=14, bold=True)

def add_meta(doc, text):
    p = doc.add_paragraph()
    p.paragraph_format.space_before = Pt(0)
    p.paragraph_format.space_after = Pt(10)
    r = p.add_run(text)
    set_font(r, size_pt=9, color=(100,100,100))

def add_h2(doc, text):
    """トピック見出し（H2相当）"""
    p = doc.add_paragraph()
    p.paragraph_format.space_before = Pt(10)
    p.paragraph_format.space_after = Pt(3)
    r = p.add_run(text)
    set_font(r, size_pt=12, bold=True)

def add_speaker(doc, speaker):
    """話者名（第1レベル、インデント）"""
    p = doc.add_paragraph(style='Normal')
    p.paragraph_format.space_before = Pt(3)
    p.paragraph_format.space_after = Pt(1)
    p.paragraph_format.left_indent = Pt(14)
    r = p.add_run(speaker)
    set_font(r, size_pt=10, bold=True)
    return p

def add_item(doc, label, content):
    """ラベル: 内容（第2レベル、インデント）"""
    p = doc.add_paragraph(style='Normal')
    p.paragraph_format.space_before = Pt(1)
    p.paragraph_format.space_after = Pt(1)
    p.paragraph_format.left_indent = Pt(28)
    if label:
        r_label = p.add_run(f"{label}: ")
        set_font(r_label, size_pt=10, bold=True)
    r_content = p.add_run(content)
    set_font(r_content, size_pt=10)
    return p

def add_question(doc, content):
    """Q：質問内容（第1レベル、インデント。「Q：」のみbold）"""
    p = doc.add_paragraph(style='Normal')
    p.paragraph_format.space_before = Pt(3)
    p.paragraph_format.space_after = Pt(1)
    p.paragraph_format.left_indent = Pt(14)
    r_label = p.add_run("Q：")
    set_font(r_label, size_pt=10, bold=True)
    r_content = p.add_run(content)
    set_font(r_content, size_pt=10)
    return p

# ドキュメント生成
os.makedirs(os.path.dirname(OUTPUT_PATH), exist_ok=True)
doc = Document()
set_narrow_margins(doc)
set_default_font(doc)

# タイトル・メタ
add_title(doc, "〔タイトル〕")
add_meta(doc, "取材日：〔日付〕　形式：〔オンライン/対面〕\n媒体：〔媒体名〕　取材対象：〔氏名〕　聞き手：〔ライター名〕")

# トピック1
add_h2(doc, "1. 〔トピック名〕")
add_question(doc, "〔質問の要点〕")
add_speaker(doc, "〔話者名〕")
add_item(doc, "〔ラベル〕", "〔発言内容の要約〕")
add_item(doc, "〔ラベル〕", "〔発言内容の要約〕")

doc.save(OUTPUT_PATH)
print(f"保存完了: {OUTPUT_PATH}")
```

**スクリプト生成時の注意：**
- インタビュアーの質問は `add_question(doc, "〔質問の要点〕")` で1行にまとめる。「Q：」のみboldで、回答者ブロックの前に配置する
- 実行前に `pip install python-docx --break-system-packages` でライブラリを確認する
- 出力先パスはユーザーに確認するか、文字起こしと同じフォルダの「文字起こし・構造化データ」ディレクトリに保存する

#### docxスタイル仕様

| 要素 | 関数 | サイズ | スタイル |
|---|---|---|---|
| 文書タイトル | add_title | 14pt | bold |
| メタ情報 | add_meta | 9pt | グレー(100,100,100) |
| トピック見出し | add_h2 | 12pt | bold、前後にスペース |
| 話者名 | add_speaker | 10pt | bold・第1レベルインデント（Pt(14)） |
| 質問行 | add_question | 10pt | 「Q：」のみbold・第1レベルインデント（Pt(14)） |

# プロット作成（plot-creator）

ライターが切り口と大まかな流れを入力し、AIが案件フォルダから情報を読み取ってエピソードを配置、プロットを作成する。

## インプット

【必須】
- 文字起こし原文
- 案件フォルダ（企画書・構造化ノートを含む）
- 媒体名

【ライターが入力する：STEP 1で受け取る】
- 切り口の3要素（テーマ／視点／伝えたいこと）
- 大まかな流れ（自由記述）

【ライターが必要に応じて補足】
- 企画書にない注意事項／避けたい表現／強調したいニュアンス

## 処理手順

### STEP 0：媒体プロファイルと案件フォルダを読む（必須）

媒体プロファイルから読むもの。
- 媒体のコンセプト・目的・読者層
- 記事タイプの分類（QA形式・対談・地の文など）
- 避けるべきトーン・NGワード
- 文字数目安・セクション構成のパターン

案件フォルダから読むもの。
- 企画書（トーン・読者像・媒体特有の制約）
- プロジェクトレファレンス（媒体トンマナ・禁止表現）
- 構造化ノート（全体像）
- 文字起こし原文（発言ニュアンス・細部）
- 媒体ログ・案件ログ（あれば、前回までの差分学習結果）

媒体名が指定されていない、または案件フォルダにアクセスできない場合はユーザーに確認する。

> STEP 0を省略しない。企画書とレファレンスから読めるものをライターに入力させない。

### STEP 1：ライターの入力を受け取る

以下のテンプレートをチャットに出してライターに入力してもらう。

```
## 切り口（3要素を入力してください）
- テーマ：〔この記事で扱う主題〕
- 視点：〔どの角度から切り込むか〕
- 伝えたいこと：〔読者に持ち帰ってほしいメッセージ〕

## 大まかな流れ
〔自由記述。記事の流れを文章で書いてください〕

## 補足（任意）
- 企画書にない注意事項：
- 避けたい表現：
- 強調したいニュアンス：
```

ライターの入力が揃うまで先に進まない。曖昧な部分があれば確認する。

**例外：自動実行モード**
`audio-transcriber` または `torimaterial-post-processing` から呼び出された場合は、ライターの入力なしに事前資料（企画概要・質問項目）から切り口を推定して暫定プロットを作成する。その際は編集メモに次を必ず記載する。
- 採用した切り口（推定であることを明記）
- 候補として検討した他の切り口（あれば1〜2案）
- 切り口選定の確信度（高／中／低）
- 「ライターの切り口確認が未取得・暫定版」である旨

### STEP 2：エピソードを構造化ノート＋文字起こし原文から拾い上げる

ライターが書いた大まかな流れに沿って、構造化ノート（全体像）と文字起こし原文（温度感・固有の言い回し・細部）の両方を参照する。セクションごとに使うエピソードと発言候補（原文引用）を整理する。
並行して、流れに入っていないが採用候補になりそうなエピソードを別途リストアップし、STEP 3のプロット末尾で提示する。採否はライターが判断する。

### STEP 3：プロットを作成する

```markdown
# プロット：〔記事タイトル案〕

## 1. 記事メタ情報
- 媒体／文字数目安／トーン要約／想定読者

## 2. 切り口（ライター入力）
- テーマ／視点／伝えたいこと

## 3. 大まかな流れ（ライター入力）

## 4. プロット本体
### 【導入部】（約〔文字数〕字）
- 見出し案／要点／使うエピソード（構造化ノート参照箇所＋原文引用）／編集メモ
### 【プロフィール】
### 【セクション1〜N】〔見出し案〕（導入と同じ項目）
### 【締めセクション】〔見出し案〕

## 5. AIからのエピソード追加提案
- 候補ごとに：該当箇所の原文引用／配置案／理由

## 6. 全体への補足メモ
- トーンの注意点／全体構造の意図／トンマナ／前回の差分学習からの参照事項
```

### STEP 4：プロットをライターに提示する

採用候補エピソードの取り込み可否、セクション順・見出しの調整、切り口の解釈ズレを確認する。
調整指示を反映してから後工程の執筆スキル（`article-writer` または `client-a-article-generator`）に渡す。

## 注意事項

- 捏造禁止：文字起こし・構造化ノートにない情報をプロットに含めない
- 発言候補は原文ベース：引用は原文の言い回しを尊重する
- 編集判断はライター：AIは提案、最終判断はライター
- 品質チェックはしない：誤字・表記ゆれ・ファクトチェックはQCスキルの仕事
- 差分学習はしない：納品稿との差分蓄積は `diff-learning` スキル（週次の weekly-diff-learning-scan が呼ぶ）が担当する

| 別系統（週次） | `diff-learning`（差分学習） |

パターン②：CLIで自動化（音声から一括）

入り口は、指定のフォルダに入れた音声ファイル。置いた瞬間に、文字起こしから構成案まで一気に流れる。

流れ

  1. 指定のフォルダ（音声の投入口）に音声ファイルを入れる
  2. macOSのlaunchdがフォルダの変化を検知し、監視スクリプト（watch.sh）を起動する
  3. watch.shがClaude Code（CLI）をヘッドレスで呼び出す
  4. Gemini APIで音声を文字起こしする
  5. ファイル名またはサブフォルダ名から案件フォルダを特定し、事前資料（企画概要・質問項目）の話者名・固有名詞でGeminiの精度を補強する
  6. 整形→構造化（取材）または議事録（MTG）→プロット（取材）まで仕上げる
  7. 音源はリネームして案件フォルダ内の取材音源フォルダへ移し、完了をSlackに通知する

使用するMacの機能

  • launchd（LaunchAgent）：フォルダ監視と常駐。アプリを開いていなくても、ファイルが置かれたら自動で起動する
  • Finderのサイドバー：音声の投入口フォルダを登録して、ドラッグ＆ドロップしやすくする

作り方

  1. GEMINI_API_KEYを環境変数に設定する
  2. 監視スクリプト（watch.sh）と、処理手順を書いたスキルファイルを用意する
  3. LaunchAgentの設定ファイル（.plist）を ~/Library/LaunchAgents/ に置き、監視対象フォルダと起動スクリプトを指定して有効化する
  4. 仕上げはClaude Codeに任せる。スキルファイルをまるごと渡し、watch.shとplistの雛形まで組ませると速い

パターン②のスクリプト雛形（transcribe.py／watch.sh／plist）

自分の環境に合わせて、パス・ユーザー名・APIキーを設定してください。そのまま貼り付けても動きません。Claude Codeにこの3つを渡して「自分の環境に合わせて直して」と頼むと速いです。

① transcribe.py — Gemini文字起こしの本体。引数で動く汎用スクリプト。ほぼそのまま使える。

#!/usr/bin/env python3
"""Gemini API 音声文字起こしスクリプト

使い方:
  python3 transcribe.py <音声ファイル> <出力先.txt> <プロンプトファイル>

必要な環境変数:
  GEMINI_API_KEY
"""
import os
import sys
import time
from pathlib import Path


GEMINI_MODEL = "gemini-3-flash-preview"

MIME_MAP = {
    ".ogg": "audio/ogg",
    ".mp3": "audio/mpeg",
    ".mp4": "audio/mp4",
    ".m4a": "audio/mp4",
    ".wav": "audio/wav",
    ".flac": "audio/flac",
    ".aac": "audio/aac",
    ".webm": "audio/webm",
}


def get_mime_type(file_path: str) -> str:
    ext = Path(file_path).suffix.lower()
    return MIME_MAP.get(ext, "audio/mpeg")


def transcribe(audio_path: str, output_path: str, prompt_path: str) -> None:
    api_key = os.environ.get("GEMINI_API_KEY")
    if not api_key:
        print("エラー: GEMINI_API_KEY が設定されていません。", file=sys.stderr)
        print("設定方法: export GEMINI_API_KEY='your-key'", file=sys.stderr)
        sys.exit(1)

    try:
        from google import genai
    except ImportError:
        print("エラー: google-genai が未インストールです。", file=sys.stderr)
        print("インストール: pip install google-genai", file=sys.stderr)
        sys.exit(1)

    audio_path = str(Path(audio_path).expanduser().resolve())
    if not Path(audio_path).exists():
        print(f"エラー: 音声ファイルが見つかりません: {audio_path}", file=sys.stderr)
        sys.exit(1)

    prompt_text = Path(prompt_path).read_text(encoding="utf-8")
    client = genai.Client(api_key=api_key)

    print(f"[1/3] 音声ファイルをアップロード中: {Path(audio_path).name}", flush=True)
    mime_type = get_mime_type(audio_path)
    uploaded = client.files.upload(file=audio_path, config={"mime_type": mime_type})

    # ファイルが処理完了するまで待機
    print("[2/3] Gemini で文字起こし生成中...", flush=True)
    retries = 0
    while True:
        state = str(getattr(uploaded, "state", "")).upper()
        if "FAIL" in state:
            print(f"エラー: ファイルアップロード失敗 (state={state})", file=sys.stderr)
            sys.exit(1)
        if "PROCESSING" not in state:
            break
        if retries > 30:
            print("エラー: アップロードタイムアウト（60秒超過）", file=sys.stderr)
            sys.exit(1)
        time.sleep(2)
        retries += 1
        uploaded = client.files.get(name=uploaded.name)

    response = client.models.generate_content(
        model=GEMINI_MODEL,
        contents=[uploaded, prompt_text],
    )

    print(f"[3/3] 結果を保存中: {output_path}", flush=True)
    Path(output_path).parent.mkdir(parents=True, exist_ok=True)
    Path(output_path).write_text(response.text, encoding="utf-8")

    # アップロードしたファイルを削除（48時間後に自動削除されるが念のため）
    try:
        client.files.delete(name=uploaded.name)
    except Exception:
        pass

    print(f"完了: {output_path}")


if __name__ == "__main__":
    if len(sys.argv) != 4:
        print("使い方: transcribe.py <音声ファイル> <出力先.txt> <プロンプトファイル>")
        print("例:     transcribe.py audio.mp3 /tmp/output.txt /tmp/prompt.txt")
        sys.exit(1)
    transcribe(sys.argv[1], sys.argv[2], sys.argv[3])

② audio-transcriber-watch.sh — フォルダ監視からClaude Code起動までの接着役。$HOME 相対だが、CLAUDE_BIN は自分の claude の場所に合わせる。

#!/bin/bash
# audio-transcriber-watch.sh
# launchd の WatchPaths から _inbox を監視し、音声ファイルを検知したら
# Claude CLI で audio-transcriber スキルを自動実行する。

WORK_DIR="$HOME/Desktop/WORK"
INBOX_DIR="$HOME/audio-inbox"
LOCKFILE="/tmp/audio-transcriber-watch.lock"
LOG_FILE="$HOME/Library/Logs/audio-transcriber.log"
CLAUDE_BIN="/usr/local/bin/claude"

# ---- ログ関数 ------------------------------------------------
log() { echo "$(date '+%Y-%m-%d %H:%M:%S') $*" >> "$LOG_FILE"; }

# ---- 起動確認（デバッグ用） ----------------------------------
log "WatchPaths 発火（INBOX=$INBOX_DIR）"

# ---- 音声ファイル確認 ----------------------------------------
# .m4a を追加（iPhoneボイスメモ等に対応）
AUDIO=$(find "$INBOX_DIR" -maxdepth 1 \
    \( -name "*.mp3" -o -name "*.ogg" -o -name "*.mp4" -o -name "*.m4a" \) \
    2>/dev/null | head -1)
log "検索結果: ${AUDIO:-なし}"

# 対象ファイルがなければ静かに終了（WatchPaths は他の変化でも発火する）
[ -z "$AUDIO" ] && exit 0

# ---- 同時実行防止 --------------------------------------------
if [ -f "$LOCKFILE" ]; then
    log "実行中のセッションあり、スキップ: $(basename "$AUDIO")"
    exit 0
fi
touch "$LOCKFILE"
trap "rm -f '$LOCKFILE'" EXIT

# ---- ファイル転送完了を待つ（5秒後にサイズ変化がなければ進む）------
SIZE1=$(stat -f%z "$AUDIO" 2>/dev/null)
sleep 5
SIZE2=$(stat -f%z "$AUDIO" 2>/dev/null)
if [ "$SIZE1" != "$SIZE2" ]; then
    log "ファイル転送中、スキップ: $(basename "$AUDIO")"
    exit 0
fi

log "音声ファイル検知: $(basename "$AUDIO")"

# ---- macOS 通知 -----------------------------------------------
osascript -e "display notification \"$(basename "$AUDIO")\" with title \"Audio Transcriber\" subtitle \"文字起こしを開始します\"" 2>/dev/null

# ---- ログウィンドウを自動表示 ----------------------------------
osascript -e "tell application \"Terminal\" to do script \"tail -f '$LOG_FILE'\"" -e "tell application \"Terminal\" to activate" 2>/dev/null

# ---- PATH と GEMINI_API_KEY をログインシェルから取得 -----------
export PATH="/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin:$PATH"
if [ -z "$GEMINI_API_KEY" ]; then
    export GEMINI_API_KEY=$(zsh -c \
        '. ~/.zshenv 2>/dev/null; . ~/.zprofile 2>/dev/null; . ~/.zshrc 2>/dev/null; echo $GEMINI_API_KEY' \
        2>/dev/null)
fi

if [ -z "$GEMINI_API_KEY" ]; then
    log "エラー: GEMINI_API_KEY が未設定。~/.zshrc に export GEMINI_API_KEY='...' を追加してください"
    osascript -e "display notification \"GEMINI_API_KEY が未設定です\" with title \"Audio Transcriber\" subtitle \"エラー\"" 2>/dev/null
    # exit 1 → exit 0 に変更：launchd の WatchPaths 抑制を防ぐ
    exit 0
fi

# ---- プロンプトをファイルに書き出す ----------------------------
SKILL_MD="$HOME/Library/Scripts/audio-transcriber-SKILL.md"
TRANSCRIBE_PY="$HOME/Library/Scripts/transcribe.py"
PROMPT_FILE="/tmp/audio-transcriber-prompt.txt"
{
    cat "$SKILL_MD"
    printf '\n\n---\n## 今すぐ実行してください\n\n'
    printf '対象音声ファイル: %s\n' "$AUDIO"
    printf 'transcribe.py のパス: %s\n' "$TRANSCRIBE_PY"
    printf 'GEMINI_API_KEY は環境変数にセット済みです。\n\n'
    printf 'Step 2（案件フォルダ特定）から最後まで完走してください。確認なしで処理を進めてください。\n'
    printf '\n## ⚠️ 実行上の制約（必ず守ること）\n\n'
    printf '%s\n' '- 完了報告を出力したら即座に終了すること。それ以降のシェルコマンドや追加処理は一切実行しない'
    printf '%s\n' '- `claude` コマンドをサブプロセスとして呼び出さない（`claude --print` 等の再帰呼び出し禁止）'
    printf '%s\n' '- `at` コマンドを使用しない（macOS通知のスケジュールも含む）'
    printf '%s\n' '- `launchctl`・`cron`・`osascript` など通知・スケジュール系コマンドを実行しない'
    printf '%s\n' '- スキルの呼び出しはインライン実行（Read + 手順を自分で実行）のみ。サブエージェント起動禁止'
} > "$PROMPT_FILE"
log "プロンプト文字数: $(wc -c < "$PROMPT_FILE")"

# ---- Claude CLI 実行（プロンプトをパイプで渡す）----------------
log "Claude CLI 起動中..."
cd "$HOME" && cat "$PROMPT_FILE" | "$CLAUDE_BIN" \
    --dangerously-skip-permissions \
    --model claude-sonnet-4-6 \
    --print \
    >> "$LOG_FILE" 2>&1
EXIT_CODE=$?

if [ $EXIT_CODE -eq 0 ]; then
    log "完了"
    # ---- サマリーファイルを読んで通知文を組み立てる ----------------
    SUMMARY_FILE="/tmp/audio-transcriber-summary.txt"
    if [ -f "$SUMMARY_FILE" ]; then
        CHARS=$(grep '^CHARS=' "$SUMMARY_FILE" | cut -d= -f2)
        UNCLEAR=$(grep '^UNCLEAR=' "$SUMMARY_FILE" | cut -d= -f2)
        RATIO=$(grep '^RATIO=' "$SUMMARY_FILE" | cut -d= -f2)
        CHARS_FMT=$(printf "%'d" "$CHARS" 2>/dev/null || echo "$CHARS")
        # 200字/分未満 → 大幅省略の可能性あり
        LOW_RATIO=0
        if [ -n "$RATIO" ] && [ "$RATIO" -gt 0 ] 2>/dev/null && [ "$RATIO" -lt 200 ]; then
            LOW_RATIO=1
            log "警告: 文字起こし密度が低い (${RATIO}字/分、期待値 300字/分以上)"
        fi
        if [ "${UNCLEAR:-0}" -ge 5 ] || [ "$LOW_RATIO" -eq 1 ]; then
            RATIO_NOTE=""
            [ "$LOW_RATIO" -eq 1 ] && RATIO_NOTE=" / 密度${RATIO}字/分"
            SUBTITLE="⚠️ 完了 | ${CHARS_FMT}文字 / [聞き取り不明] ${UNCLEAR}箇所${RATIO_NOTE}"
        else
            SUBTITLE="✅ 完了 | ${CHARS_FMT}文字 / [聞き取り不明] ${UNCLEAR}箇所"
        fi
        rm -f "$SUMMARY_FILE"
    else
        SUBTITLE="文字起こし完了"
    fi
    osascript -e "display notification \"$(basename "$AUDIO")\" with title \"Audio Transcriber\" subtitle \"$SUBTITLE\"" 2>/dev/null
else
    # Claude がエラー終了した場合もログ・通知して exit 0 で終了
    # → launchd に exit 0 を返すことで WatchPaths 抑制を防ぐ
    log "エラー終了 (exit=$EXIT_CODE) ※次回の自動起動は維持されます"
    osascript -e "display notification \"ログを確認してください (exit=$EXIT_CODE)\" with title \"Audio Transcriber\" subtitle \"エラーが発生しました\"" 2>/dev/null
fi

# launchd の WatchPaths 抑制を防ぐため、常に exit 0 で終了する
exit 0

構造化ノートとは

構造化ノートは、文字起こしをトピック×話者で整理し直した中間資料。

## トピック：創業の経緯
- 田中氏：2018年に前職の同僚と起業。きっかけは顧客から…
- 佐藤氏：当初は受託開発が中心。転機は2021年の…
## トピック：現在の課題
- 田中氏：採用。特にミドル層の…

1時間の取材の文字起こしは2〜3万字になるが、構造化ノートなら数分で全体を把握でき、制作中にあの話はどこで出たかを探す時間がなくなる。入れ忘れや処理失敗で未処理の素材が残っても、毎朝の自動処理が案件フォルダを巡回して拾うため、取りこぼしも防げる。

8-4. 原稿作成

プロット→初稿→修正の順。各段階の要点。

プロット（起点は人間の切り口）

切り口と大まかな流れを人間が入力し、AIが構造化ノートと文字起こし原文からエピソードを拾い上げてプロットに組み立てる。プロットで決めるのは、媒体と全体の文字数目安、セクション数と各セクションの文字数配分、各セクションに入れる内容と使うエピソード、記事全体のテーマと読後感、書かずにカットする内容、取材先の意向や表現上の配慮といった注意点。文字数とセクション構成を先に固めておくと、初稿が分量・構成のブレなく出る。プロットの精度が初稿の精度を決める、編集者の仕事の中心になる工程。

執筆

まず長文を一気に書き出させて全体の流れを整え、そのあとパート（セクションやQAのひとかたまり程度）ごとに区切って修正する。順序は必ず全体から部分へ。最初から部分を磨くと流れが歪む。執筆時には媒体レファレンス（9章）が自動で読み込まれ、媒体のトンマナが効いた初稿が出る。

執筆を担う article-writer スキルの中身。媒体名はサンプルに置き換えてある。

# 記事原稿作成スキル

プロット（構成案）と文字起こし整理結果・企画概要をもとに、媒体特性に合わせた原稿を執筆するスキル。対応媒体は `_references/[slug]/profile.md` を自動読み込みして媒体ルールを適用する。未対応媒体は企画書・参考記事から媒体ルールを推定して執筆する。

## 対応媒体と媒体プロファイル

媒体名を受け取ったら、下表の slug に対応する `_references/[slug]/profile.md` を読み込み、`_references/[slug]/references/` 配下のすべての参照資料も読み込む。以降の執筆はプロファイルの指示が最優先される。

| ライターが伝える媒体名 | slug |
|---|---|
| サンプル1媒体 | `sample1` |
| サンプル2媒体 | `sample2` |

profile内に `references/file.md` の表記がある場合は `_references/[slug]/references/file.md` として解決する。

媒体名がいずれにも該当しない場合は、profile読み込みをスキップし、企画書・過去記事・プロジェクトレファレンスから媒体ルールを推定する（後述のステップ1-1）。

## 入力要件

以下の素材を最初にすべて提供してもらう。不足している場合は執筆を始めず、ユーザーに確認する。

### 必須
- **企画書**：媒体情報・読者層・文字数・記事形式・トンマナ・禁止表現などを含むもの
- **文字起こし**：transcript-structurizer で整理済みのもの
- **プロット**：plot-creatorで作成済みのもの。6セクション構成（1. 記事メタ情報／2. 切り口／3. 大まかな流れ／4. プロット本体／5. AIからのエピソード追加提案／6. 全体への補足メモ）が前提

### 強く推奨
- **差分学習ログ**：媒体ログ（`_references/[媒体名]/差分学習ログ.md`）と案件ログ（案件フォルダ内 `差分学習ログ.md`）。あれば過去のパターンを反映できる
- **過去記事のURLまたはPDF**：媒体のトンマナ・文体・構成パターンを把握するために使用。複数あれば複数提供が望ましい
- **プロジェクトレファレンス**：媒体トンマナ・禁止表現のドキュメント
- **その他注意事項**：表記ルール、NGワード、特記事項など

### 企画書からの読み取り項目

企画書が提供された場合、以下を読み取り媒体ルールとして適用する。企画書の指定がデフォルトルールと異なる場合は企画書を優先する。

- 記事形式（QA / 第三者記述 / 座談会 / ルポ）
- 文字数（全体・各セクション）
- 文体（です・ます調 / だ・である調）
- 読者層・トンマナ
- 禁止表現・注意事項

> プロット外枠・差分学習ログ・媒体プロファイルの所在・受け渡しパスの正式定義は `_references/_interface.md` を参照。
> 本文と差異がある場合は仕様書を正とする。ただしプロット本体の章立て・トンマナは各媒体プロファイルを正とする。

> 実行モデル：メインスレッドで実行する。サブエージェントに委譲しない
> （切り口の解釈・文体の判断は執筆モデルの品質が必要）。

## 処理手順

本スキルは文字起こし整理・プロット作成・初稿執筆を一気通貫で実行する。

### ステップ0：素材確認と媒体プロファイル読み込み

#### 0-1. 素材の確認

必須素材（企画書・文字起こし）が揃っているか確認する。不足している場合は執筆を始めず、ユーザーに確認する。

#### 0-0. プロット承認ゲート（最初に確認）

プロットを受け取った場合、承認ステータスを確認する。「暫定（未承認）」または承認ステータスの記載がない場合は執筆を開始せず、以下をライターに求める。

```
プロットの承認ステータスが「暫定（未承認）」または未記載です。
執筆前に切り口の3要素（テーマ／視点／伝えたいこと）をご確認ください。
確認・承認後に執筆を開始します。
```

ライターから承認の意思表示（「OK」「進めて」「確認した」など）を得てから執筆に進む。

#### 0-2. 媒体プロファイルの読み込み（対応媒体の場合）

媒体名が「対応媒体と媒体プロファイル」表にある場合は以下を実行する。

1. `_references/[slug]/profile.md` を Read で読み込む
2. `_references/[slug]/references/` 配下を Glob で一覧化し、profile に明記された参照ファイルおよび記事タイプに応じたサンプル記事を Read で読み込む（例：サンプル1媒体なら記事タイプを判定して該当サンプル記事を読む）
3. profile の「フォーマット」「文体・トンマナ」「文字数目安」「チェックリスト」を以降の執筆の最優先ルールとする

未対応媒体の場合はスキップしてステップ1へ進む。

### ステップ1：事前準備

#### 1-1. 企画書・過去記事・プロジェクトレファレンスから読み取り

媒体プロファイルを読み込んだ場合でも、企画書に固有の指定があればそれを優先する（特定回の特例・取材対象固有の注意事項など）。媒体ルールと企画書ルールがコンフリクトした場合はユーザーに確認する。

企画書と過去記事（URLまたはPDF）、プロジェクトレファレンスから以下を把握する：

- **記事形式**：企画書に明記されている場合はそれに従う。明記がない場合は下記の形式から判定する
  - **形式A（QA形式）**：質問と回答の対話形式。「――質問」「話者名：回答」の構成。インタビュー記事に多い
  - **形式B（第三者記述）**：ライターが第三者視点で記述し、発言を「」で引用。企業紹介・事例記事に多い
  - **形式C（座談会）**：複数話者の会話形式。「名字「発言」」の構成。座談会・パネル記事に多い
  - **形式D（ルポ）**：情景描写を交えたライター視点の記述。現場ルポ・体験記事に多い
- **文字数目安**（全体・各セクション）
- **文体**（です・ます調 / だ・である調）
- **読者層**・**トンマナ**
- **禁止表現・注意事項**
- 過去記事がある場合：文体の雰囲気・見出しの付け方・引用の入れ方・締めのパターン

#### 1-2. 差分学習ログを読む

過去のパターンが蓄積されていれば、執筆判断に反映する。

- **媒体ログ**：`_references/[媒体名]/差分学習ログ.md` を読む。媒体ごとのトンマナ・構成パターン・締め方の傾向を把握
- **案件ログ**：案件フォルダ内 `差分学習ログ.md` を読む。取材対象者のクセ・記事固有の調整事項を把握

両方のログから、以下を抽出して執筆時に反映する。

- 構成の変更パターン：過去の案件でプロットからどう構成が変わったか
- エピソードの取捨選択パターン：採用したが結果的にカットされたエピソード、追加採用されたエピソードの傾向
- 表現の変更パターン：トーン調整・言い換え・カットの傾向
- 着地の変更パターン：締めのトーンやメッセージの強度の調整傾向

ログがない、または項目が空の場合はスキップする。

#### 1-3. プロットを読み込む

plot-creatorで作成されたプロットの6セクションを読み込む。

1. **記事メタ情報**：媒体・文字数目安・トーン要約・想定読者
2. **切り口**：テーマ／視点／伝えたいこと（ライターの3要素入力）
3. **大まかな流れ**：ライターが書いた流れの自由記述
4. **プロット本体**：各セクションの「見出し案」「要点」「使うエピソード」「文字起こし原文の引用」「編集メモ」
5. **AIからのエピソード追加提案**：採否がライターによって判断済みのもの（採用したものだけ反映）
6. **全体への補足メモ**：トーンの注意点・全体構造の意図・トンマナ・前回の差分学習からの参照事項

執筆中はこのプロットを唯一のガイドラインとする。逸脱したい場合は先にユーザーに確認する。

### ステップ2：原稿執筆（初稿）

ステップ1で判定した記事形式とプロットに沿って執筆する。

#### 形式A：QA形式（インタビュー記事）

質問と回答の対話形式。読者がインタビューの場に同席しているような臨場感。

```markdown
## セクション見出し

――質問文？

**話者名（以下、略称）**：回答本文。回答本文。

――次の質問文？

**略称**：回答本文。
```

**適用場面**: オウンドメディアのインタビュー、専門家への取材、対談記事
**特徴**: 質問の前に「――」（ダッシュ）を置く。話者名は初出フルネーム、以降略称。

#### 形式B：第三者記述形式

ライターが第三者視点で記述し、要所で発言を「」引用する形式。

```markdown
## セクション見出し

本文テキスト。本文テキスト。「引用コメント」と○○氏は語る。本文テキスト。

小見出し

本文テキスト。○○氏は「引用コメント」と説明する。
```

**適用場面**: 企業紹介、人物紹介、事例記事、紙媒体の記事
**特徴**: 地の文が主体。コメントは本文の3〜4割程度が目安。

#### 形式C：座談会形式

複数の話者が会話する形式。話者の発言を「」で囲み、名前を冒頭に記載。

```markdown
## セクション見出し

＞質問文

氏名（以下、名字）「発言内容。発言内容」

名字「発言内容。発言内容」

＞次の質問文

名字「発言内容」
```

**適用場面**: 座談会、パネルディスカッション、複数人の対話記事
**特徴**: 話者間の掛け合いを活かす。話者のバランスに配慮。

#### 形式D：ルポ・エッセイ形式

ライターの視点で現場の様子や体験を描写しつつ、取材内容を織り交ぜる形式。

```markdown
## セクション見出し

情景描写や状況説明。取材対象の○○さんは「引用コメント」と語る。

続きの描写。具体的なエピソード。
```

**適用場面**: 現場ルポ、体験記事、紀行文、文化紹介
**特徴**: 情景描写が多い。コメント引用は控えめ。

プロットの構成に沿って、以下の順序で執筆する。

#### 3-1. 導入部（リード文）

**書く内容：**
- プロット本体の【導入部】に書かれた「要点」「使うエピソード」「文字起こし原文の引用」「編集メモ」を反映する
- テーマの背景・社会的意義・問題提起
- 読者の関心を引くフック
- 登場人物・取材対象の紹介
- 「この記事を読むとわかること」の示唆

**注意：**
- 導入部の文字数はプロット【導入部】の指定または記事メタ情報の文字数目安に従う
- 指定がない場合は200〜400字を目安に
- 読者が「続きを読みたい」と思える内容にする
- 媒体のトンマナに合わせた文体で書き出す

#### 3-2. 本文セクション

**各セクションの書き方：**
- プロット本体の「見出し案」をもとに見出しを確定
- プロット本体の「要点」（2〜3行）でそのセクションのねらいを把握
- プロット本体の「使うエピソード」と「文字起こし原文の引用」を本文に展開
- プロット本体の「編集メモ」で構成意図・補足が必要な箇所を確認
- AIからのエピソード追加提案でライターが採用したものがあれば、配置案に沿って組み込む
- 具体例・エピソード・数字を盛り込み、抽象的にならないようにする

**セクション間のつながり：**
- 前のセクションの内容を受けて自然に次へ移行
- 唐突な話題転換を避ける（つなぎの一文を入れる）
- 見出しだけ読んでも記事の流れがわかるようにする

**見出しのルール：**
- 見出しは「そのセクションに何が書いてあるか」が読者に伝わる実務的なトーンにする
- コピー的・抽象的な見出しにしない
- 見出しだけ追えば記事全体の展開がわかるように

#### 3-3. 質問の設計（QA形式の場合）

質問は以下の3つの機能を意識して設計する：

1. **読者の感情・疑問を代弁する**：読者が感じるであろう驚きや共感を質問に込める（例：「『チームの一員』ですか。かなり距離感が近いですね」）
2. **次の発言を自然に引き出す**：前の回答を受けた自然な流れで次の話題に誘導する
3. **記事のメッセージを補強する**：質問を通じて、記事全体のメッセージが強まるように設計する（例：「実際にサポートを受けていかがでしたか」で顧客満足を引き出す）

**注意：**
- 質問を増やしてセクションを刻みすぎない。回答の中に複数の情報を自然に織り込む方が、会話の流れが自然になる
- 反応型の質問（「〜ですか。それはすごいですね」のように驚きを表現）も効果的

#### 3-4. 締め

**書く内容：**
- プロット本体の【締めセクション】の「要点」「使うエピソード」「編集メモ」を反映する
- 切り口の「伝えたいこと」を回収するメッセージの着地
- 将来展望、読者へのメッセージ、行動喚起など

**注意：**
- 最終セクション内に含めるか、独立段落にするかはプロット・媒体の指定に従う
- 新しい情報を締めで出さない（まとめに徹する）
- プロットで指定された余韻のつくり方を尊重する

## 執筆ルール

### 文体の基本

| 項目 | ルール |
|------|--------|
| 敬体/常体 | 企画概要の指定に従う。指定なしの場合「です・ます調」 |
| 引用 | 話者の発言は「」で囲む |
| 専門用語 | 初出時に平易な言葉で補足。読者層に合わせて調整 |
| 段落の長さ | 1段落150〜300字程度。長すぎる段落は分割 |

### 引用（コメント）のルール

- 文字起こしの発言を**そのまま使える場合は原文を活かす**
- 話し言葉の整え方は媒体のトンマナに合わせる：
  - カジュアル系 →「〜なんです」「〜ですよね」を適度に残す
  - フォーマル系 →「〜です」「〜ました」に整える
- 冗長な部分は要約してよいが、**ニュアンスを損なわない**
- 発言の意味を変える改変は禁止
- 相槌・間投詞（「えーと」「まあ」等）は原則カット

### 話者名の表記

| タイミング | 表記 |
|-----------|------|
| 初出 | フルネーム＋敬称（「○○ ○○氏」「○○ ○○さん」等） |
| 2回目以降 | 略称（「○○氏」「○○さん」等） |
| 企業への言及 | 初出フル表記、以降「同社」「同団体」等 |

※敬称のルール（氏/さん/役職名）は媒体の指定に従う。指定がない場合は「さん」。

### 見出しの付け方

- **具体的に**: 内容が推測できる見出し（「取り組み」→「創業100年の酒蔵が挑む海外展開」）
- **内容の要約**: そのセクションに何が書いてあるかが読者に伝わること。コピー的・抽象的にしない
- **簡潔に**: 20字前後が目安（媒体の指定があればそちらに従う）
- **記事の流れが見える**: 見出しだけ追えば記事全体の展開がわかるように
- **「——」使用禁止**: 見出し・リード・本文を問わず「——」を絶対に使わない。「——」が許可されるのはQA形式の質問行冒頭（`――質問文？` の形式）のみ。本文やリードで使いたくなった箇所は、接続詞・句読点・文の分割など自然な文章表現で書き直す

### 数字の表記

- 媒体の指定に従う（全角指定 / 半角指定）
- 指定がない場合は半角を基本とする
- 年号・日付・金額・数量は統一して表記

## 注意事項

8-5. 校正

校正はひとつにまとめず、観点を分けて独立に実行する。一度に全部を見させると見落とすため、観点ごとにスキルを使い分ける。

観点 内容 注意
誤字脱字・言い回し 表記ルールを読み込ませた状態でチェック 表記ルールの事前整備が前提
ファクト 固有名詞・肩書・数字・日付を文字起こし・一次情報と突き合わせ 原稿を書いたスレッドとは別スレッド、できれば別AIで行う（下の注記参照）
捏造・脚色 言っていないことを足していないか、発言の意図を変えていないか ネガティブ表現の前向きな言い換え、わかりやすくする補足・整理は通常の編集の範囲
差分（任意） バージョン間の差分を確認し、指示していない箇所の書き換えを検出 マストではない。ファイルが分かれているときや、先祖返り（古い版に戻ってしまう）のリスクがあるときに行う。AIは指示外の箇所を善意で直すことがある

使うスキルは観点ごとに分かれている。

  • 表記・誤字脱字・不自然な言い回しは、最終チェックスキルでまとめて検出し、修正案を出させる
  • ファクトの照合は、ファクトチェックスキルで原稿の固有名詞・数字・発言を文字起こしと突き合わせる。文字起こしに載っていない事実は、ネット検索で一次情報・公開情報とも突き合わせる
  • 納品前にまとめてかけるなら、品質チェック（QC）チームのスキル。編集・トンマナ確認→事実確認→表記校正→SEOチェックの順に通し、指摘をQCレポート（Word）にまとめてくれる

どのスキルも出すのは指摘と修正案まで。採否と最終判断は人間が行う。指摘を原稿に反映したら、反映漏れを防ぐため、指摘に出た語句をWordやドキュメントの検索機能で1つずつ拾って直す。

書いた本人に採点させない。 ファクトと捏造のチェックは、原稿を書いたスレッドとは別スレッド、できれば別AIで実行する。同じスレッドのまま確認させると、見落としをそのまま引き継いでしまう。

校正・ファクトチェックを担う2スキルの中身。

# 原稿最終チェックスキル

> 実行モデル：定型処理モデル（CLAUDE.mdの役割対応表を参照）のサブエージェントに委譲して実行する。
> メインスレッドでは実行しない。結果はファイルまたは要約でメインスレッドに返す。

## 概要

原稿完成後の最終チェックを行うスキル。誤字脱字、表記ゆれ、不自然な言い回しを検出し、具体的な修正案を提示する。`article-writer`（全媒体対応）や `client-a-article-generator`（クライアントA専用）などの原稿作成スキルと組み合わせて使用可能。

## ワークフロー

### ステップ1：チェック対象の確認

**原稿の取得:**
- 現在のチャット内に原稿がある場合 → そのまま使用
- 原稿がない場合 → ユーザーに提示を依頼

**参照資料の確認（任意）:**
- 文字起こしがチャット内にある場合 → 固有名詞・数字の照合に使用
- 過去のチャットに文字起こしがある場合 → `recent_chats`で検索して取得可能

### ステップ2：誤字脱字チェック

以下の観点で原稿全体をスキャンする。

**チェック項目:**
1. **タイプミス・変換ミス**
   - 同音異義語の誤用（例：「意外」⇔「以外」「成果」⇔「生果」）
   - 脱字（助詞の欠落、送り仮名の欠落）
   - 衍字（余分な文字の混入）

2. **固有名詞の誤り**
   - 人名、社名、製品名のスペルミス
   - 文字起こしがある場合は照合して確認

3. **数字・単位の誤り**
   - 桁数の間違い
   - 単位の誤記（万/億、%など）

### ステップ3：表記ゆれチェック

原稿内の表記統一を確認する。

**チェック項目:**

1. **数字の表記**
   - 全角/半角の混在（例：「１０個」と「10個」）
   - 漢数字/算用数字の混在（例：「三つ」と「3つ」）

2. **漢字/ひらがな/カタカナ**
   - 同一語の表記揺れ（例：「できる」⇔「出来る」「ない」⇔「無い」）
   - 外来語のカタカナ表記（例：「コンピュータ」⇔「コンピューター」）

3. **送り仮名**
   - 送り仮名の揺れ（例：「行なう」⇔「行う」「落し」⇔「落とし」）

4. **人名・社名の呼称**
   - 初出と2回目以降の表記揺れ（例：初出「田中氏」、以降「田中さん」）
   - 敬称の統一（氏/さん/様）

5. **記号・句読点**
   - 括弧の種類（「」⇔『』⇔（）⇔""）
   - 句読点の統一（、。⇔，．）

### ステップ4：不自然な言い回しチェック

読みやすさ・自然さの観点でチェックする。

**チェック項目:**

1. **文の構造**
   - 主語と述語の不一致・ねじれ
   - 修飾語と被修飾語の距離が遠い
   - 一文が長すぎる（目安：80字以上）

2. **表現の重複**
   - 同じ言葉の繰り返し（例：「〜することができることになった」）
   - 意味の重複（例：「まず最初に」「約〜くらい」「一番最後」）

3. **文末表現**
   - 同じ文末の連続（例：「〜だ。〜だ。〜だ。」が5回以上続く場合など、明らかに単調な箇所）
   - ※「ですます調」ベースの原稿でも、リズムをつけるために「〜する。」「〜だ。」などの言い切り表現を意図的に混ぜることがある。これは誤りではないため指摘不要

4. **接続・論理**
   - 接続詞の不適切な使用
   - 話の流れが唐突または飛躍している箇所

5. **インタビュー原稿特有の確認**
   - 発言者の言い回しが原稿の文体から浮いていないか
   - 引用と地の文の区別が明確か
   - **「——」使用禁止**：見出し・リード・本文を問わず「——」が含まれていないか全文確認する。「——」が使えるのはQA形式の質問行冒頭（`――質問文？` の形式）のみ。それ以外の箇所に使われていたら必ず指摘し、文章を自然につなぐ表現（接続詞・句読点・文の分割など）への書き直しを提案する

### ステップ5：文字起こしとの整合性確認（該当する場合）

チャット内に文字起こしがある場合、以下を確認する。

**確認項目:**
- 固有名詞（人名、社名、地名、製品名）の正確性
- 数字（金額、年号、比率など）の正確性
- 発言内容の主旨が正しく反映されているか

### ステップ6：指摘事項の出力

発見した問題を以下の形式で出力する。

**出力形式:**

```

| No. | 該当箇所 | 問題 | 修正案 |
|-----|----------|------|--------|
| 1 | 「〇〇」 | （問題の説明） | 「〇〇」 |
| 2 | 「〇〇」 | （問題の説明） | 「〇〇」 |

## 2. 表記ゆれ
| No. | 表記A | 表記B | 推奨表記 | 該当箇所 |
|-----|-------|-------|----------|----------|
| 1 | 「〇〇」 | 「〇〇」 | 「〇〇」 | （箇所を明示） |

## 3. 不自然な言い回し
| No. | 該当箇所 | 問題 | 修正案 |
|-----|----------|------|--------|

|-----|------------|------------------|--------|
| 1 | 「〇〇」 | 「〇〇」 | 「〇〇」 |

（全体的な評価、特に注意すべき点があれば記載）
```

## チェックの優先度

問題の重大性に応じて優先度をつける：

1. **高**：事実誤認、固有名詞の誤り、意味が変わる誤字
2. **中**：表記ゆれ、文法的な誤り
3. **低**：軽微な言い回し、スタイルの好みに関わる指摘

## 注意事項

- 問題のない箇所は指摘不要。問題があった箇所のみ報告
- 「好みの問題」レベルの軽微な指摘は控えめに
- 原稿の文体・トーンは変えない（修正案は原稿のスタイルに合わせる）
- 指摘が多い場合は、重要度の高いものから順に提示
- 問題がなければ「問題なし」と報告し、必要以上にコメントしない
- **文体の意図的な混在は許容する**：「ですます調」の原稿でも、リズムや強調のために「〜する。」「〜だ。」などの言い切り表現を混ぜることは一般的な技法。これを誤りとして指摘しない

## 媒体別の表記ルール参照

原稿作成時に使用した媒体固有のスキルがある場合、そのスキルの表記ルールに従う：
- **サンプル媒体**：数字は全角、NGワード確認など
- **その他媒体**：各スキルの references/ 内の表記ルールを参照

# Interview Fact Checker

> 実行モデル：定型処理モデル（CLAUDE.mdの役割対応表を参照）のサブエージェントに委譲して実行する。
> メインスレッドでは実行しない。結果はファイルまたは要約でメインスレッドに返す。

## Overview

このスキルはインタビューの文字起こしと作成中の原稿の整合性を確認する。ユーザーが原稿の特定箇所について「文字起こしではどう言っていたか」を確認したい場合に、過去のチャットから文字起こしを参照し、該当箇所をピックアップして原稿との比較を行う。

## Workflow

### Step 1: 文字起こしの取得

ユーザーが現在のチャット内で文字起こしをアップロードしていない場合、過去のチャットから文字起こしを検索する。

**検索方法:**
- `recent_chats`ツールで最近のチャットを取得(n=10程度)
- 文字起こしファイルがアップロードされているチャットを特定
- 該当する文字起こしが見つからない場合は、ユーザーに文字起こしのアップロードを依頼

**現在のチャット内に文字起こしがある場合:**
- 過去のチャット検索は不要。直接Step 2へ進む

### Step 2: 該当箇所の検索とピックアップ

ユーザーが確認したい内容について、文字起こしから該当箇所を検索する。

**検索のポイント:**
- ユーザーが指定したキーワードやトピックを元に検索
- 該当する発言を複数見つかった場合は、すべてリストアップ
- 発言者(インタビュイー)の名前も含めて抽出
- 前後の文脈も含めて引用(発言の意図を正確に把握するため)

**出力形式:**
```
【発言者名】
「(該当する発言を文字起こしからそのまま引用)」
```

### Step 3: 発言の主旨を簡潔にまとめる

ピックアップした発言内容を分析し、主旨を簡潔にまとめる。

**まとめ方:**
- 発言の核心的なメッセージを1〜2文で要約
- ニュアンスや文脈を考慮
- 事実関係(数字、固有名詞、時期など)を正確に抽出

**出力形式:**
```
【発言の主旨】
(簡潔なまとめ)
```

### Step 4: 原稿との整合性確認

ユーザーが提示した原稿の該当箇所と、文字起こしの発言内容を比較する。

**確認項目:**
1. **事実関係の正確性**
   - 数字、固有名詞、時期、場所などが一致しているか
   - 具体的なエピソードや事例が正確に記載されているか

2. **ニュアンスの一致**
   - 発言の意図や感情が適切に反映されているか
   - 強調点や重要なポイントが原稿でも強調されているか
   - 肯定的/否定的なトーンが変わっていないか

3. **文脈の正確性**
   - 発言の前後関係が正しく理解されているか
   - 条件や前提が省略されていないか

**出力形式:**
```
【整合性確認結果】
✓ 問題なし / ⚠ 要確認 / ✗ 齟齬あり

(具体的な指摘)
```

### Step 5: 修正案の提示(齟齬がある場合のみ)

整合性確認で齟齬が見つかった場合、具体的な修正案を提示する。

**修正案の作成方針:**
- 文字起こしの発言内容に忠実に
- 原稿の文体やトーンを維持
- 必要最小限の修正(全体を書き直さない)

**出力形式:**
```
【修正案】
■ 現在の原稿
「(問題のある箇所を引用)」

■ 修正案
「(修正後の文案)」

■ 修正理由
(なぜこの修正が必要か、文字起こしのどの部分と齟齬があるかを説明)
```

## 注意事項

- 文字起こしは「発言のまま」を引用する(意訳しない)
- 原稿との比較では、表現の違いだけでなく、意味やニュアンスの違いに注意
- 軽微な言い回しの違いは指摘不要。重要な事実やニュアンスの齟齬のみ指摘
- 複数の齟齬がある場合は、優先度の高いものから順に提示
- 修正案は原稿の文体に合わせて提案する

8-6. 納品後

納品稿と初稿の差分は学習材料として蓄積する。仕組みは9-4で詳述。

8-7. CMS入力（半自動化・任意）

公開用CMSへの入力も、コネクタやブラウザ操作で半自動化できる。媒体ごとに入力欄（見出し・本文・タグ・画像）が決まっているため、原稿mdから各項目を割り当てて下書き登録まで任せ、最終確認と公開ボタンだけ人間が担う運用にできる。同じ媒体を継続して受けている案件ほど効果が出やすい。

9. 媒体ルール・トンマナをAIに再現させる（媒体レファレンス）

媒体レファレンスは、媒体ごとのルールをまとめた資料一式。AIに「この媒体らしさ」を再現させる要で、執筆スキルとセットで機能する。

9-1. 構成

WORK/
└── _references/
    └── 月刊ダミーズ/

        ├── references/（参考記事2〜3本）
        └── 差分学習ログ.md
  • 置き場所はWORKフォルダ配下の _references/。案件フォルダ（4章）と同じWORKの中にあり、媒体ごとの正本をここにまとめる

  • ローカルが正本。週次のスケジュールタスクでGoogleドライブへ自動バックアップする

項目 内容 記載例

  • プロファイルと参考記事はセットで機能する
  • 注意点・NG・トンマナは、最初は経験や編集部の指摘から書く。運用後は、差分学習ログ（9-4）で確度が高くなったパターンを昇格させて足していく

9-3. つくり方の手順

ゼロから手書きする必要はない。

  1. その媒体の過去記事3〜5本をAIに読ませ、プロファイルの下書きを生成させる
  2. 人間が修正する。特にNG事項と注意点は経験からしか書けない部分
  3. 使いながら更新する

最初の精度は7割で十分。運用で育てるほうが、最初に完璧を目指すより早く品質が安定する。

  • 案件をこなすほど初稿の精度が上がる。媒体レファレンスは書いて終わりの設定ではなく、運用で育てる資産

10. ルーティンワークを自動で回す（スケジュールタスクと日々の運用）

10-1. スケジュールタスクとは

スケジュールタスク（scheduled tasks）は、指定した時刻・周期でClaudeが自動で作業を実行する仕組み。Coworkの「scheduled」がこれにあたる。人がいない時間に定型処理を回す、制作フロー自動化の土台。

  • 設定は言葉で頼むだけ。「毎朝8時に今日の予定をまとめてSlackに送って」で登録される。プログラミングや専用設定の知識は不要
  • 周期実行（毎朝、毎週月曜、毎時など）と単発予約（明日9時に1回だけ）の両方に対応
  • 確認・変更も言葉で行う。「定期タスクを一覧して」「毎朝のタスクを8時半に変更して」
  • 実行結果はSlack通知などで受け取る設計にする。画面を見張る必要がない

10-2. 運用しているスケジュールタスクの例

タイミング 処理内容 通知
毎朝 カレンダー読み取り→当日のスケジュールメモとリマインド作成 Slack
毎朝 未処理の取材素材を案件フォルダから探して後処理 Slack

  • 向かない仕事：人の判断・承認が途中に必要なもの。これはパイプライン（7-4）で人間の判断ポイントを挟んで実行する

10-3. メンテナンスの習慣

スキル・フォルダ構成・レファレンスは、放っておくと少しずつズレていく。設定とスキルがズレると自動処理が静かに壊れ、出力の精度も落ちる。だからメンテは地味で面倒でも、組んだ自動化を生かし続けるための必須作業。面倒な分、日々のメンテ自体もできるだけタスク化して、スケジュールタスクで自動で回す。

主なメンテと自動化の可否。

  • スキルの同期：Cowork上とローカルのスキルをそろえる。片方だけ新しいと挙動がズレるので、スケジュールタスクで定期実行にする
  • スキル台帳の更新：スキルの一覧と依存関係を最新に保つ。これもスケジュールタスクで定期的に回せる
  • スレッドの引き継ぎ：重くなったら引き継ぎメモを作って新スレッドへ移る。ここは人の判断（長いスレッドは精度も速度も落ちる）

10-4. 運用設定のコツ

  • アラート・承認メッセージは日本語にさせる
  • 削除など破壊的な操作の前は、影響範囲を先に説明させる
  • 承認の求めすぎは作業の邪魔になる。読み取り系は承認なし、書き込み・削除だけ確認、のバランスにする
  • PowerPoint・デザイン系の生成はトークン消費が早い。使いどころを選ぶ
  • 重要データのバックアップは常時とる。AIは稀にファイルを壊す
  • 途中生成物はmdで残す（4-4参照）。事故が起きても途中から再開できる

11. AIで原稿をつくるときの注意点

実務で確認された、AIの出力との付き合い方。

注意点 起きること 対策
キャッチーなネタへの偏り わかりやすい数字や強いネタにとびつき、記事がチープになる 引きの強さと記事の質を分けて評価する
文章のクセ・文体の崩れ 主語と述語がねじれる、適切な接続詞が使えない、同じ表現や語尾を多用する、ダッシュ「―」を使いがち、である調とですます調が混ざる、「」を付けすぎる 文体・表記ルールを設定とレファレンスに書いておき、最終チェックスキルで拾って直す
文字数が数えられない 「2000字で書いた」という申告と実数がずれる 申告を信じず、必ずツールで実測する
要約系のブレ 要素が多くアウトプットが短い作業（1時間取材→500字など）は出力が安定しない 選択肢が膨大なため。人間の編集視点で選ぶ前提で使う
自己評価が甘い 「問題ありません」と答えるが見落としがある チェックは別スレッド・別AIで行う
長スレッドでの劣化 スレッドが長くなるとこれまでの経緯を忘れ、精度・速度も落ちる 引き継ぎメモを作って新スレッドへ
固有名詞・数字の誤り 名前、肩書、数字、日付を取り違える 一次情報との突き合わせを習慣化する
詰め込みすぎの劣化 一度に多くを頼むと全体の質が落ちる 工程を分けて、ひとつずつ渡す
  • 補足：論理的なもの・パターンが決まっているものはブレが小さい。要約・抽出・選択の比重が大きい仕事ほどブレが大きく、編集判断の価値が高い
  • 文体・表記のような決まったルールは、毎回言わずに設定かスキルに書き込む（2章・7章参照）。同じ指摘を繰り返さずに済む
  • 一方、出力の正しさを詰めたいときは「本当にそれでいいの？」「見落としはない？」と問い返すと、AIが再点検して精度が上がる。決まったルールは仕組み化、判断の詰めは問い返し、と使い分ける
